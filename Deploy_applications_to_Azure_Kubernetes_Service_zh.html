將 AI 推論 API 部署到 Azure Kubernetes Service

在本練習中，您將部署 Azure 資源，包括 Microsoft Foundry AI 模型、Azure Container Registry (ACR) 和 Azure Kubernetes Service (AKS) 叢集。接著，您會完成 Kubernetes 資訊清單檔案，以定義容器規格、健康情況探查、資源限制和負載平衡。將容器化的 API 部署到 AKS 後，您會使用 Python 用戶端應用程式來測試已部署的 API 端點，包括健康情況檢查、就緒狀態驗證，以及 AI 模型推論要求。

本練習執行的工作：

  • 下載專案入門檔案
  • 將資源部署到 Azure
  • 完成 deployment.yaml 和 service.yaml 檔案，並將容器部署到 AKS
  • 執行用戶端應用程式以測試 API

本練習大約需要 30 分鐘完成。

重要： Azure Container Registry 工作執行目前不支援使用 Azure 免費額度。本練習需要 Pay-As-You-Go 或其他付費方案。

開始之前

若要完成此練習，您需要：

  • 具備部署所需 Azure 服務權限的 Azure 訂用帳戶。如果您還沒有，可以 註冊一個。
  • 安裝於其中一個 支援的平台 上的 Visual Studio Code。
  • 最新版的 Azure CLI。
  • Kubernetes 命令列工具， kubectl。
  • 選用： Python 3.12 或更新版本。

下載專案入門檔案並部署 Azure 服務

在本節中，您會下載主控台應用程式的入門檔案，並使用指令碼將所需服務部署到您的 Azure 訂用帳戶。Azure Managed Redis 部署需要 5 到 10 分鐘完成。

  1. 開啟瀏覽器並輸入下列 URL 以下載入門檔案。檔案會儲存在您的預設下載位置。

    https://github.com/MicrosoftLearning/mslearn-azure-ai/raw/main/downloads/python/aks-deploy-python.zip

  2. 將檔案複製或移動到系統中您想要進行此專案作業的位置，然後將檔案解壓縮到資料夾中。

  3. 啟動 Visual Studio Code (VS Code)，並在選單中選取 File > Open Folder... ，然後選擇包含專案檔案的資料夾。

  4. 此專案同時包含 Bash (azdeploy.sh) 和 PowerShell (azdeploy.ps1) 的部署指令碼。開啟適合您環境的檔案，並變更指令碼頂端的兩個值以符合您的需求，然後儲存變更。 注意： 不要變更指令碼中的其他任何內容。

    "<your-resource-group-name>" # Resource Group name
"<your-azure-region>" # Azure region for the resources

    注意： 建議使用下列三個 Azure 區域之一進行部署： eastus2、 swedencentral，或 australiaeast。這些區域支援部署本練習所使用的 AI 推論模型。

  5. 在功能表列中選取 Terminal > New Terminal ，以在 VS Code 中開啟終端機視窗。

  6. 執行下列命令登入您的 Azure 帳戶。依照提示選取本練習要使用的 Azure 帳戶和訂用帳戶。

    az login

  7. 執行下列命令，確保您的訂用帳戶具有安裝 AKS、ACR 和 Foundry AI 模型所需的資源提供者。 Microsoft.Compute、 Microsoft.Network以及 Microsoft.Storage 提供者是 AKS 的相依項目，Azure 通常會自動註冊它們，但明確註冊可避免新訂用帳戶偶爾發生叢集建立失敗。

    az provider register --namespace Microsoft.CognitiveServices
az provider register --namespace Microsoft.ContainerService
az provider register --namespace Microsoft.ContainerRegistry
az provider register --namespace Microsoft.Compute
az provider register --namespace Microsoft.Network
az provider register --namespace Microsoft.Storage

  8. 確認您位於專案根目錄中，然後在終端機中執行適當的命令以啟動部署指令碼。

    Bash

    bash azdeploy.sh

    PowerShell

    ./azdeploy.ps1

    注意： 如果 PowerShell 因為指令碼未經數位簽章而封鎖它，請在同一個終端機工作階段中執行下列命令，然後再次執行部署指令碼。此命令只會變更目前 PowerShell 處理程序的執行原則。

    Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass

將資源部署到 Azure

在部署指令碼執行期間，請依照下列步驟在 Azure 中建立所需的資源。

  1. 輸入 1 以啟動 1. Provision gpt-5-mini model in Microsoft Foundry 選項。此選項會在資源群組尚不存在時建立它、在 Microsoft Foundry 中建立資源，並將 gpt-5-mini 模型部署到該資源。

    重要： 如果模型部署期間發生錯誤，請輸入 2 以啟動 2. Delete/Purge Foundry deployment 選項。這會刪除部署並清除此資源名稱。退出選單，並將部署指令碼中的區域變更為其他建議區域之一。然後重新啟動部署指令碼，再次執行模型佈建選項。

  2. 模型部署完成後，輸入 3 以啟動 3. Create Azure Container Registry (ACR)。這會建立儲存 API 容器的資源，之後會將其提取到 AKS 資源中。

  3. ACR 資源建立完成後，輸入 4 以啟動 Build and push API image to ACR。此選項會使用 ACR 工作來建置映像並將其加入 ACR 存放庫。此作業可能需要 3 到 5 分鐘完成。

  4. 映像已建置並推送至 ACR 後，輸入 5 以啟動 5. Create AKS cluster 選項。這會建立以受控識別設定的 AKS 資源，並授與該服務從 ACR 資源提取映像的權限。此作業可能需要 5 到 10 分鐘完成。

  5. AKS 資源部署完成後，輸入 6 以啟動 6. Check deployment status 選項。此選項會回報三個資源中的每一個是否都已成功部署。

    如果所有服務都傳回 successful 訊息，請輸入 8 以退出部署指令碼。

接下來，您要完成將 API 部署到 AKS 所需的 YAML 檔案。

完成 YAML 部署檔案並部署到 AKS

在本節中，您將完成 deployment.yaml 和 service.yaml 檔案。Deployment 資訊清單定義 API 容器如何在 AKS 中部署與管理，而 Service 資訊清單則透過負載平衡器將 API 對外公開。

  1. 開啟 k8s/deployment.yaml 檔案以開始完成內容。

  2. 找到 # BEGIN: Container specification 註解，並在其下方將下列 YAML 區段加入資訊清單。請確保 YAML 縮排正確。

    containers:  # List of containers to run in the pod
- name: api
  image: ACR_ENDPOINT/aks-api:latest  # Container image from ACR
  imagePullPolicy: Always  # Always pull the latest image from registry
  ports:  # Ports exposed by the container
  - name: http
    containerPort: 8000
    protocol: TCP

    本節定義容器規格，包括要從 ACR 使用的容器映像、提取原則，以及容器為 HTTP 流量公開的連接埠。

  3. 找到 # BEGIN: Liveness Probe Configuration 註解，並在其下方將下列 YAML 區段加入資訊清單。請確保 YAML 縮排正確。

    livenessProbe:  # Detects if container is alive or needs restart
  httpGet:
    path: /healthz  # Health check endpoint path
    port: http
  initialDelaySeconds: 10  # Seconds to wait before first check
  periodSeconds: 30
  timeoutSeconds: 5
  failureThreshold: 3  # Consecutive failures before restarting container

    本節設定存活探查，它會定期向 /healthz 端點發出 HTTP 要求，以檢查容器是否健康。如果探查連續失敗三次，Kubernetes 會自動重新啟動容器。

  4. 找到 # BEGIN: Resource Limits Configuration 註解，並在其下方將下列 YAML 區段加入資訊清單。請確保 YAML 縮排正確。

    resources:  # CPU and memory resource specifications
  requests:  # Minimum resources guaranteed to the container
    memory: "256Mi"
    cpu: "250m"
  limits:  # Maximum resources the container can use
    memory: "512Mi"
    cpu: "500m"

    本節定義容器的 CPU 和記憶體資源。Requests 指定保證提供的最小資源，而 limits 設定容器可使用的最大資源。這有助於 Kubernetes 有效率地排程 pod，並避免資源飢餓。

  5. 儲存您的變更，並花幾分鐘檢視已完成的 deployment.yaml 檔案。

接下來，您要更新 service.yaml 檔案。

  1. 開啟 k8s/service.yaml 以完成此檔案。

  2. 將下列 YAML 加入資訊清單。請確保 YAML 縮排正確。

    apiVersion: v1
kind: Service  # Service: exposes pods on a network and provides load balancing
metadata:
  name: aks-api-service  # Unique name for the service
  labels:
    app: aks-api # Matches deployment and pod labels
  annotations:
    service.beta.kubernetes.io/azure-load-balancer-internal: "false"  # Use public load balancer
spec:  # Service specification
  type: LoadBalancer  # Exposes service externally
  selector:  # Selects which pods to route traffic to based on labels
    app: aks-api
    version: v1
  ports:  # Port mappings between service and pods
  - name: http
    port: 80  # Service port exposed externally
    targetPort: http  # Pod container port to forward traffic to
    protocol: TCP
  sessionAffinity: None  # Client requests not pinned to specific pods

    此資訊清單會建立一個 LoadBalancer Service，透過 Azure Load Balancer 將您的 API pod 對外公開。它會將連接埠 80 的傳入流量路由到容器的連接埠 8000，並使用標籤選取器來識別哪些 pod 應接收流量。

  3. 儲存您的變更，並花幾分鐘檢視此檔案。

將資訊清單套用至 AKS

在本節中，您會使用部署指令碼將資訊清單套用到 AKS。

  1. 確認您位於專案根目錄中，然後在終端機中執行適當的命令以啟動部署指令碼。

    Bash

    bash azdeploy.sh

    PowerShell

    ./azdeploy.ps1

    注意： 如果 PowerShell 因為指令碼未經數位簽章而封鎖它，請在同一個終端機工作階段中執行下列命令，然後再次執行部署指令碼。此命令只會變更目前 PowerShell 處理程序的執行原則。

    Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass

  2. 輸入 7 以啟動 7. Deploy to AKS 選項。此選項會執行多項作業：擷取您的 AKS 認證並設定 kubectl，將 Cognitive Services OpenAI User 角色指派給 AKS kubelet 受控識別，讓 API 可以使用 Microsoft Entra ID 驗證至 Foundry，更新部署資訊清單中的 ACR 端點與 Foundry 端點，然後使用 kubectl apply 將兩個資訊清單部署到您的 AKS 叢集。作業完成後，請輸入 8 以退出部署指令碼。

  3. 在終端機中執行下列命令以驗證部署。預期 kubectl get deploy,svc 會顯示 Deployment 的 READY 為 1/1 （或您的複本數），而 Service 的 EXTERNAL-IP 具有公用 IP（不是 <pending>）。當更新完成時，rollout 命令應輸出 deployment "aks-api" successfully rolled out 。

    kubectl get deploy,svc
kubectl rollout status deploy/aks-api

執行用戶端應用程式

在本節中，您會設定 Python 環境，然後使用用戶端應用程式對 API 執行操作

設定 Python 環境

在本節中，您將建立 Python 環境並安裝相依性。

  1. 請確認您在終端機中位於專案的 client 資料夾。

  2. 在 VS Code 終端機中執行下列命令以建立 Python 環境。

    python -m venv .venv

  3. 執行下列命令以啟用 Python 環境。 注意： 在 Linux/macOS 上，使用 Bash 命令。在 Windows 上，使用 PowerShell 命令。如果在 Windows 上使用 Git Bash，請使用 source .venv/Scripts/activate。

    Bash

    source .venv/bin/activate

    PowerShell

    .\.venv\Scripts\Activate.ps1

  4. 在 VS Code 終端機中執行下列命令以安裝相依性。

    pip install -r requirements.txt

使用應用程式執行操作

現在該執行用戶端應用程式，對 API 執行各種操作了。此應用程式提供選單驅動式介面。

  1. 在終端機中執行下列命令以啟動主控台應用程式。如有需要，請先參考本練習前面提供的命令來啟用環境，再執行此命令。

    python main.py

  2. 輸入 1 以啟動 1. Check API Health (Liveness) 選項。這會驗證 API 容器正在執行，並且會回應健康情況檢查；這也是 Kubernetes 存活探查所使用的相同端點。

  3. 輸入 2 以啟動 2. Check API Readiness (Foundry Connectivity) 選項。這會確認 API 能成功連線到 Foundry 模型端點，並已準備好處理推論要求。

  4. 輸入 3 以啟動 3. Send Inference Request 選項。這會將單一提示傳送至 API，並從已部署的模型接收完整回應。單次推論要求適合批次處理、自動化工作，或當您需要一次取得整個回應以供進一步處理時。

  5. 輸入 4 以啟動 4. Start Chat Session (Streaming) 選項。這會啟動互動式聊天工作階段，模型的回應會在產生時即時串流顯示。

完成後，請輸入 5 以退出應用程式。

清除資源

現在您已完成練習，應刪除所建立的雲端資源，以避免不必要的資源使用。

  1. 在 VS Code 終端機中執行下列命令以刪除資源群組及群組內的所有資源。將 <rg-name> 替換為您在本練習稍早選擇的名稱。此命令會在 Azure 中啟動背景工作以刪除資源群組。

    az group delete --name <rg-name> --no-wait --yes

注意： 刪除資源群組會刪除其中包含的所有資源。如果您為本練習選擇了現有的資源群組，則本練習範圍以外的任何現有資源也會一併刪除。

疑難排解

如果您在完成本練習時遇到問題，請嘗試下列疑難排解步驟：

驗證 Azure 資源部署

  • 瀏覽至 Azure portal 並找到您的資源群組。
  • 確認 Microsoft Foundry 資源的 Provisioning State 為 Succeeded 且 gpt-5-mini 模型已部署。
  • 確認 Azure Container Registry (ACR) 存在，且包含 aks-api 映像。
  • 檢查 AKS 叢集是否處於 Succeeded 狀態，且節點正在執行。

驗證 AKS 部署狀態

  • 執行 kubectl get pods 以檢查 API pod 是否正在執行。請查看 Running 狀態。
  • 執行 kubectl get svc 以確認 LoadBalancer Service 已指派外部 IP（不是 <pending>）。
  • 執行 kubectl describe pod <pod-name> 以在發生問題時查看詳細的 pod 狀態與事件。
  • 使用 kubectl logs <pod-name> 檢查 pod 記錄，以查看容器啟動錯誤或執行階段問題。

驗證 YAML 檔案完整性

  • 確認所有 YAML 區段都已正確加入 deployment.yaml 和 service.yaml 中相應的註解標記之間。
  • 確認 YAML 縮排正確（使用空格，不是定位字元），因為縮排錯誤會導致部署失敗。
  • 確認部署指令碼已正確將 ACR 端點替換到部署資訊清單中。

驗證用戶端設定

  • 檢查 .env 檔案是否存在於 client 資料夾中，並包含有效的 API_ENDPOINT 值。
  • 確認 API 端點使用的是 LoadBalancer Service 的正確外部 IP。
  • 確認您可以透過在終端機中執行 curl http://<external-ip>/healthz 來連線到 API 端點。

檢查 Python 環境與相依性

  • 確認在執行用戶端應用程式之前已啟用虛擬環境。
  • 確認 requirements.txt 中的所有套件都已成功安裝；請執行 pip list 進行確認。
  • 確認您是從 client 目錄執行用戶端。
