使用 Azure Event Grid 發佈與接收事件

AI 內容審核系統在分類與檢閱提交內容時會產生大量事件。Azure Event Grid 提供路由層，根據事件類型將這些事件導向正確的下游取用者，因此每個處理常式都只會收到所需事件，而不必輪詢或手動篩選。

在此練習中，您將部署具有 namespace topic 與篩選事件訂用帳戶的 Event Grid Namespace，然後建置一個 Python Flask 應用程式，以發佈內容審核事件並使用 pull delivery 接收它們。Event Grid 訂用帳戶會將已標記內容、已核准內容及所有事件路由至不同的訂用帳戶，讓您能觀察篩選在實務上的運作方式。您也會使用 pull delivery 提供的 receive、acknowledge 與 reject 作業來控制應用程式處理事件的方式。

此練習中執行的工作：

  • Download the project starter files
  • 部署包含 namespace topic 的 Event Grid Namespace
  • 建立具有類型篩選器的事件訂用帳戶
  • Add code to the starter files to complete the app
  • 執行應用程式以發佈、接收及處理審核事件

此練習大約需要 30 分鐘完成。

加入發佈審核事件的程式碼

在本節中，您將加入程式碼，將五個內容審核事件發佈到 Event Grid namespace topic。這些事件使用 CloudEvents v1.0 結構描述，並代表不同的審核結果——已標記內容、已核准內容與升級審查——讓您觀察每個 subscription 的 event type filter 如何決定它傳遞哪些事件。

此函式會從 moderation_events.json 檔案載入事件定義，其中包含 CloudEvent envelope 欄位（type, source, subject）以及 data 承載。發佈時，函式會為每個事件加入唯一的 id 以及目前 UTC timestamp ，接著建立 CloudEvent 物件，並以 send() 方法在單一要求中發佈它們。 EventGridPublisherClient 是以 namespace_topic 參數建構，以指向 namespace topic endpoint，並使用 DefaultAzureCredential 作為 Microsoft Entra 驗證。

  1. Locate the # BEGIN PUBLISH EVENTS FUNCTION comment and add the following code under the comment. Be sure to check for proper code alignment.

    def publish_moderation_events():
    """Publish content moderation events to the Event Grid namespace topic."""
    client = get_publisher_client()
    results = []

    # Load event definitions from the JSON file. Each entry contains the
    # CloudEvent envelope fields (type, source, subject) and the data
    # payload that mirrors a realistic AI content moderation pipeline.
    json_path = os.path.join(os.path.dirname(__file__), "moderation_events.json")
    with open(json_path, "r") as f:
        event_definitions = json.load(f)

    # Build CloudEvent objects from the definitions, adding a unique id
    # and a current UTC timestamp to each event at publish time.
    events = []
    for defn in event_definitions:
        defn["data"]["timestamp"] = datetime.now(timezone.utc).isoformat()
        events.append(
            CloudEvent(
                type=defn["type"],
                source=defn["source"],
                subject=defn["subject"],
                data=defn["data"],
                id=str(uuid.uuid4())
            )
        )

    # send() publishes all events to the Event Grid namespace topic in a
    # single request. Event Grid then evaluates each subscription's
    # filters and routes matching events to the configured subscriptions.
    client.send(events)

    for event in events:
        results.append({
            "content_id": event.data["contentId"],
            "event_type": event.type.split(".")[-1],
            "category": event.data["category"],
            "confidence": event.data["confidence"],
            "status": "published"
        })

    return results

  2. Save your changes and take a few minutes to review the code.

加入接收與確認事件的程式碼

在本節中，您將加入程式碼，從每個 subscription 接收事件並加以確認，以驗證篩選是否正常運作。Pull delivery 表示您的應用程式會連線到 Event Grid 並主動要求事件，而不是由 Event Grid 將事件推送到端點。每個收到的事件都包含一個 lock token，您必須加以確認，才能將該事件從 subscription 中永久移除；否則事件會在鎖定時間到期後重新傳遞。

此函式會建立 EventGridConsumerClient ，對應三個 subscriptions。 receive() 方法會傳回 ReceiveDetails 物件清單，每個物件都包含 CloudEvent (.event）以及含有 lock_token的 broker properties。處理後，函式會呼叫 acknowledge() ，並傳入收集到的 lock tokens，以確認事件已成功處理。

  1. Locate the # BEGIN CHECK DELIVERY FUNCTION comment and add the following code under the comment. Be sure to check for proper code alignment.

    def check_filtered_delivery():
    """Receive and acknowledge events from each subscription to verify filtering."""
    flagged = []
    approved = []
    all_events = []

    # Receive from the sub-flagged subscription, which only delivers
    # events where the event type is com.contoso.ai.ContentFlagged.
    # receive() returns a list of ReceiveDetails, each containing
    # the CloudEvent and a lock token for acknowledgment.
    consumer = get_consumer_client(SUB_FLAGGED)
    details = consumer.receive(max_events=10, max_wait_time=10)
    tokens = []
    for detail in details:
        event = detail.event
        flagged.append({
            "content_id": event.data.get("contentId"),
            "category": event.data.get("category"),
            "severity": event.data.get("severity"),
            "confidence": event.data.get("confidence")
        })
        tokens.append(detail.broker_properties.lock_token)
    # acknowledge() removes the events from the subscription so they
    # are not delivered again on the next receive call.
    if tokens:
        consumer.acknowledge(lock_tokens=tokens)

    # Receive from the sub-approved subscription, which only delivers
    # events where the event type is com.contoso.ai.ContentApproved.
    consumer = get_consumer_client(SUB_APPROVED)
    details = consumer.receive(max_events=10, max_wait_time=10)
    tokens = []
    for detail in details:
        event = detail.event
        approved.append({
            "content_id": event.data.get("contentId"),
            "category": event.data.get("category"),
            "severity": event.data.get("severity"),
            "confidence": event.data.get("confidence")
        })
        tokens.append(detail.broker_properties.lock_token)
    if tokens:
        consumer.acknowledge(lock_tokens=tokens)

    # Receive from the sub-all-events subscription, which has no filter
    # and delivers every event published to the topic (audit log).
    consumer = get_consumer_client(SUB_ALL)
    details = consumer.receive(max_events=10, max_wait_time=10)
    tokens = []
    for detail in details:
        event = detail.event
        all_events.append({
            "content_id": event.data.get("contentId"),
            "event_type": event.data.get("modelName", "unknown"),
            "category": event.data.get("category"),
            "confidence": event.data.get("confidence")
        })
        tokens.append(detail.broker_properties.lock_token)
    if tokens:
        consumer.acknowledge(lock_tokens=tokens)

    return {
        "flagged": flagged,
        "approved": approved,
        "all_events": all_events
    }

  2. Save your changes and take a few minutes to review the code.

加入檢查並拒絕事件的程式碼

在本節中，您將加入程式碼，發佈單一測試事件、接收它、檢查完整的 CloudEvent envelope，然後拒絕它。拒絕事件表示告知 Event Grid 該事件無法處理。這與 acknowledging 不同；acknowledging 是用來確認處理成功。被拒絕的事件會被捨棄，或在有設定時移至 dead-letter 目的地。

此函式會先使用 EventGridPublisherClient 來發佈測試事件，因此無論先前事件是否已被確認，都一定會有可用事件。接著它會從 sub-flagged subscription 接收事件，擷取 CloudEvent 屬性與 broker properties（包括 delivery_count），並呼叫 reject() 並傳入 lock token。

  1. Locate the # BEGIN INSPECT AND REJECT FUNCTION comment and add the following code under the comment. Be sure to check for proper code alignment.

    def inspect_and_reject():
    """Publish one event, receive it, inspect the CloudEvent envelope, then reject it."""
    publisher = get_publisher_client()

    # Publish a single test event so there is always something to inspect,
    # regardless of whether the student already acknowledged earlier events.
    test_event = CloudEvent(
        type="com.contoso.ai.ContentFlagged",
        source="/services/content-moderation",
        subject="/content/text/test-inspect",
        data={
            "contentId": "test-inspect",
            "contentType": "text",
            "modelName": "text-moderator-v2",
            "modelVersion": "2.4.0",
            "confidence": 0.76,
            "category": "misinformation",
            "severity": "medium",
            "reviewRequired": True,
            "timestamp": datetime.now(timezone.utc).isoformat()
        },
        id=str(uuid.uuid4())
    )
    publisher.send([test_event])

    # Receive from the sub-flagged subscription to pick up the test event.
    consumer = get_consumer_client(SUB_FLAGGED)
    details = consumer.receive(max_events=1, max_wait_time=10)

    if not details:
        return None

    detail = details[0]
    event = detail.event
    lock_token = detail.broker_properties.lock_token
    delivery_count = detail.broker_properties.delivery_count

    # Capture the full CloudEvent envelope before rejecting.
    result = {
        "specversion": "1.0",
        "type": event.type,
        "source": event.source,
        "subject": event.subject,
        "id": event.id,
        "time": str(event.time) if event.time else "",
        "data": event.data,
        "delivery_count": delivery_count,
        "action": "rejected"
    }

    # reject() tells Event Grid this event cannot be processed. The event
    # is moved to the dead-letter location if configured, or discarded
    # if max delivery count has been reached.
    consumer.reject(lock_tokens=[lock_token])

    return result

  2. Save your changes and take a few minutes to review the code.

Run the app

在本節中，您將執行完成的 Flask 應用程式，以發佈內容審核事件並驗證篩選訂用帳戶是否正確傳遞它們。此應用程式提供 Web 介面，讓您可以發佈事件、從篩選訂用帳戶接收並確認事件，以及檢查並拒絕事件，以探索完整的 CloudEvent 結構與 pull delivery 作業。

  1. Run the following command in the terminal to start the app. Refer to the commands from earlier in the exercise to activate the environment, if needed, before running the command. If you navigated away from the client directory, run cd client first.

    python app.py

  2. Open a browser and navigate to http://localhost:5000 to access the app.

  3. Select Publish Moderation Events 在左側面板中。這會將五個內容審核事件發佈到 Event Grid namespace topic：兩個已標記內容事件、兩個已核准內容事件，以及一個升級審查事件。右側面板中的結果會確認每個事件都已發佈，並顯示其 content ID、event type 與 category。

  4. Select Receive & Acknowledge Events 在左側面板中。這會使用 pull delivery 從三個 subscriptions 接收事件，並在處理後加以確認。請根據每個 subscription 所設定的篩選器，驗證下列傳遞行為：

    • Flagged Subscription： 應包含兩個事件，且 category 值都表示違反政策（violence 與 hate-speech）。這些是 ContentFlagged events.
    • Approved Subscription： 應包含兩個事件，且 category 都是 safe。這些是 ContentApproved events.
    • All Events Subscription： 無論類型為何，都應包含全部五個事件，作為稽核記錄。

    升級審查事件（ReviewEscalated）只會出現在 all-events subscription 中，因為 flagged 與 approved subscriptions 的篩選器都未包含此事件類型。由於事件已被確認，再次選取此按鈕時會顯示零個事件，直到您再發佈更多事件為止。

  5. Select Inspect & Reject Event 在左側面板中。這會發佈新的測試事件、從 flagged subscription 接收它，並顯示完整的 CloudEvent envelope，其中包括 delivery_count （來自 broker properties），然後拒絕該事件。此拒絕會告知 Event Grid 該事件無法處理，因此 Event Grid 會將其捨棄，或在有設定時移至 dead-letter 目的地。

疑難排解

如果您在完成此練習時遇到問題，請嘗試下列疑難排解步驟：

驗證 Event Grid Namespace 部署

  • Navigate to the Azure portal and locate your resource group.
  • 確認 Event Grid Namespace 的 Provisioning State of Succeeded.
  • 確認 namespace topic moderation-events 存在於該命名空間內。

檢查事件訂用帳戶

  • 請執行部署指令碼的狀態檢查（選項 5），確認三個事件訂用帳戶都已建立。
  • 確認訂用帳戶顯示 Succeeded status： sub-flagged, sub-approved, and sub-all-events.
  • 如果在發佈後未收到任何事件，請確認訂用帳戶是在 topic 建立之後才建立的。必要時請重新執行部署指令碼的選項 2。

檢查程式碼完整性與縮排

  • Ensure all code blocks were added to the correct sections in event_grid_functions.py between the appropriate BEGIN/END comment markers.
  • 請確認 Python 縮排一致（使用空格而非定位字元），且所有程式碼在函式內都正確對齊。
  • 請確認指定區段之外的程式碼未被意外刪除或修改。

Verify environment variables

  • Check that the .env 檔案存在於專案根目錄，且包含 EVENTGRID_ENDPOINT, EVENTGRID_TOPIC_NAME, RESOURCE_GROUP, and NAMESPACE_NAME values.
  • Ensure you ran source .env (Bash) or . ..env.ps1 (PowerShell) to load environment variables into your terminal session.
  • If variables are empty, re-run source .env (Bash) or . ..env.ps1 (PowerShell).

Check authentication

  • Confirm you are logged in to Azure CLI by running az account show.
  • 請確認 EventGrid Data Sender 與 EventGrid Data Receiver 角色已指派在該命名空間上。必要時請再次執行部署指令碼的角色指派選項（選項 3）。

檢查 Python 環境與相依性

  • 請確認在執行應用程式之前已啟用虛擬環境。
  • 請確認來自 requirements.txt 的所有套件都已透過執行 pip list.
