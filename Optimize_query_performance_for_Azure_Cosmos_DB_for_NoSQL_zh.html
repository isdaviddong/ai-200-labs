使用 Azure Cosmos DB for NoSQL 的向量索引最佳化查詢效能

在本練習中，您將比較和調整向量索引策略以最佳化查詢效能 Azure Cosmos DB for NoSQL。向量索引顯著影響搜尋品質和 Request Unit (RU） 消耗。您建立具有三種不同索引類型的容器 -flat, quantizedFlat， 和 diskANN—載入相同的樣本資料，並執行比較搜尋來衡量效能差異。這種實踐練習可以幫助您選擇適合您的索引策略 AI 應用程式的要求。

本練習執行的工作：

  • 下載專案啟動檔案並設定部署腳本
  • 部署一個 Azure Cosmos DB for NoSQL 帳戶 具有向量搜尋功能
  • 建造 Python 用於比較向量索引效能的函數
  • 建立容器 flat, quantizedFlat， 和 diskANN 索引
  • 使用以下方法測試並比較索引效能 Flask Web 應用程式

本練習大約需要 30 分鐘即可完成。

完成向量相似度搜尋腳本

在本節中，您將完成 index_functions.py 文件中新增了執行向量相似性搜尋和效能追蹤的功能。為每個容器呼叫此函數，以比較不同索引類型處理相同查詢的方式。

  1. 打開 client/index_functions.py 文件輸入 VS Code.

  2. 搜尋 BEGIN VECTOR SIMILARITY SEARCH FUNCTION 註釋並直接在註釋後面添加以下程式碼。此函數會尋找與查詢類似的文件並追蹤效能指標。

    def vector_similarity_search(
    container_name: str,
    query_embedding: list,
    top_n: int = 5
) -> dict:
    """
    Find documents most similar to the query using vector distance.

    This function performs a vector similarity search using the VectorDistance
    function and tracks the RU consumption and execution time. Results are
    ordered by distance (lowest = most similar).

    Args:
        container_name: Name of the container to search
        query_embedding: 256-dimensional query vector
        top_n: Number of results to return

    Returns:
        Dictionary containing results, ru_charge, and execution_time_ms
    """
    container = get_container(container_name)

    # Track execution time for performance comparison
    start_time = time.time()

    # The VectorDistance function calculates distance between vectors
    # Using cosine distance: 0 = identical, 2 = opposite
    # Results ordered by distance ascending (most similar first)
    query = """
        SELECT TOP @topN
            c.id,
            c.documentId,
            c.content,
            c.metadata,
            VectorDistance(c.embedding, @queryVector) AS similarityScore
        FROM c
        ORDER BY VectorDistance(c.embedding, @queryVector)
    """

    items = list(container.query_items(
        query=query,
        parameters=[
            {"name": "@topN", "value": top_n},
            {"name": "@queryVector", "value": query_embedding}
        ],
        enable_cross_partition_query=True
    ))

    end_time = time.time()
    execution_time_ms = (end_time - start_time) * 1000

    # Get RU charge from the query - note: this is approximate for multi-page results
    # For accurate RU tracking in production, use Azure Monitor
    ru_charge = 0.0
    try:
        # The last_response_headers contains the RU charge
        ru_charge = float(container.client_connection.last_response_headers.get(
            'x-ms-request-charge', 0
        ))
    except Exception:
        pass  # RU tracking may not be available in all scenarios

    results = [
        {
            "chunk_id": item["id"],
            "document_id": item["documentId"],
            "content": item["content"],
            "metadata": item["metadata"],
            "similarity_score": item["similarityScore"]
        }
        for item in items
    ]

    return {
        "results": results,
        "ru_charge": ru_charge,
        "execution_time_ms": round(execution_time_ms, 2)
    }

  3. 搜尋 BEGIN COMPARE INDEX PERFORMANCE FUNCTION 註釋並直接在註釋後面添加以下程式碼。該函數對所有三個容器執行相同的查詢並傳回比較結果。

    def compare_index_performance(
    query_embedding: list,
    top_n: int = 5
) -> dict:
    """
    Run the same vector search query against all three containers and compare performance.

    This function executes identical vector similarity searches against containers
    with different indexing strategies (flat, quantizedFlat, diskANN) to demonstrate
    the performance characteristics of each approach.

    Args:
        query_embedding: 256-dimensional query vector
        top_n: Number of results to return from each container

    Returns:
        Dictionary with results from each container including RU costs and timing
    """
    comparison = {}

    # Test each container with the same query
    for index_type, container_name in [
        ("flat", CONTAINER_FLAT),
        ("quantizedFlat", CONTAINER_QUANTIZED),
        ("diskANN", CONTAINER_DISKANN)
    ]:
        try:
            result = vector_similarity_search(container_name, query_embedding, top_n)
            comparison[index_type] = {
                "container": container_name,
                "results": result["results"],
                "ru_charge": result["ru_charge"],
                "execution_time_ms": result["execution_time_ms"],
                "result_count": len(result["results"]),
                "status": "success"
            }
        except Exception as e:
            comparison[index_type] = {
                "container": container_name,
                "results": [],
                "ru_charge": 0,
                "execution_time_ms": 0,
                "result_count": 0,
                "status": "error",
                "error": str(e)
            }

    return comparison

查看容器設定代碼

在本節中，您將回顧 setup_containers.py 建立具有不同向量索引策略的容器的腳本。向量嵌入策略（路徑、維度、資料類型和距離函數）是在容器建立時設定的，之後無法變更。然而，向量索引類型是索引策略的一部分，可以在現有容器上更新。當您變更索引類型時，Cosmos DB 在背景執行索引轉換。儘管具有這種靈活性，預先測試不同的索引類型仍然是最佳實踐：常見的方法是使用每種索引類型建立測試容器，載入代表性範例資料，並執行基準查詢來測量 RU 投入生產設定之前的成本和延遲。

  1. 打開 client/setup_containers.py 文件輸入 VS Code.

  2. 搜尋 BEGIN CREATE FLAT CONTAINER FUNCTION 評論並審查程式碼。請注意如何 flat 索引設定：

    # Flat index: exact search, compares query against all vectors
# Higher RU cost for large datasets but guaranteed best results
indexing_policy = {
    "indexingMode": "consistent",
    "automatic": True,
    "includedPaths": [
        {"path": "/*"}
    ],
    "excludedPaths": [
        {"path": "/embedding/*"}
    ],
    "vectorIndexes": [
        {
            "path": "/embedding",
            "type": "flat"
        }
    ]
}

  3. 搜尋 BEGIN CREATE QUANTIZED CONTAINER FUNCTION 評論並審查 quantizedFlat 設定：

    # QuantizedFlat index: compressed vectors for memory efficiency
# Lower memory footprint with slight accuracy trade-off
indexing_policy = {
    ...
    "vectorIndexes": [
        {
            "path": "/embedding",
            "type": "quantizedFlat"
        }
    ]
}

  4. 搜尋 BEGIN CREATE DISKANN CONTAINER FUNCTION 評論並審查 diskANN 設定：

    # DiskANN index: approximate nearest neighbor with graph-based search
# Best performance for large datasets, slight accuracy trade-off
indexing_policy = {
    ...
    "vectorIndexes": [
        {
            "path": "/embedding",
            "type": "diskANN"
        }
    ]
}

  5. 花點時間了解索引類型之間的主要區別：

    指數型 擷取方式 最適合 權衡
    flat 精確最近鄰 資料集小，精度最高 更高 RU 對於大型資料集
    quantizedFlat 壓縮精確搜索 中等資料集，記憶體效率 精度略有下降
    diskANN 近似圖搜尋 大資料集、生產 ~95% 召回率，最佳性能

摘要

在本練習中，您比較了向量索引策略 Azure Cosmos DB for NoSQL。您部署了一個 Azure Cosmos DB 帳戶 與 EnableNoSQLVectorSearch 能力和設定 Entra ID 驗證。您使用以下命令建立了三個容器 Python SDK 具有不同的向量索引類型： flat 為了精確搜索， quantizedFlat 為了提高記憶體效率，以及 diskANN 用於生產規模的近似搜尋。你建造了 Python 在追蹤時執行向量相似性搜尋的函數 RU 消耗和執行時間。你用了一個 Flask Web 應用程式來執行比較搜尋並分析效能差異。此模式可協助您選擇最適合您的索引策略 AI 基於資料集大小、準確性要求和成本限制的應用程式。

疑難排解

如果您在此練習中遇到問題，請嘗試以下步驟：

Flask app fails to start

  • 確保 Python 虛擬環境已啟動（您應該看到 (.venv) 在你的終端機提示中）
  • 確保安裝了依賴項： pip install -r requirements.txt
  • 確保透過運行載入環境變量 source .env (Bash） 或者 . ..env.ps1 (PowerShell)
  • 確保您在 client 運行時目錄 python app.py

Authentication or access denied errors

  • 確保 Entra ID 透過執行部署腳本選項設定存取權限 3
  • 驗證您的 user 兩者都有 Contributor 角色和 Cosmos DB Built-in Data Contributor 角色
  • 確保 COSMOS_ENDPOINT 在您的終端會話中設定正確

setup_containers.py fails

  • 確保 Python 虛擬環境已激活
  • 確保環境變數已設定（COSMOS_ENDPOINT, COSMOS_DATABASE)
  • 如果容器已存在，腳本將使用現有容器

Vector search returns errors

  • 驗證容器是透過執行部署腳本選項建立的 2
  • 確保在運行搜尋之前加載範例資料
  • 檢查容器是否設定了向量嵌入策略

Cosmos DB operations fail

  • 驗證宇宙 DB 帳戶 透過執行部署腳本選項已準備就緒 4
  • 確保資料庫是在部署期間建立的
  • 檢查是否 帳戶 有 EnableNoSQLVectorSearch 能力

Environment variable issues

  • 確保 .env 文件是透過運行部署腳本選項創建的 5
  • 跑步 source .env (Bash） 或者 . ..env.ps1 (PowerShell）建立新終端後
  • 透過運行驗證變數是否已設定 echo $COSMOS_ENDPOINT (Bash） 或者 $env:COSMOS_ENDPOINT (PowerShell)

Python venv activation issues

  • 在 Linux/macOS 上，使用： source .venv/bin/activate
  • 在 Windows 上 PowerShell， 使用： ..venv\Scripts\Activate.ps1
  • 如果 activate 腳本遺失，重新安裝 python3-venv 打包並重新建立 venv
