在 Azure Kubernetes Service 上疑難排解應用程式

在本練習中，您會將容器化 API 部署到 Azure Kubernetes Service (AKS)，然後診斷並解決常見的 Kubernetes 問題。您將使用 kubectl 命令來識別問題、檢查 Pod 狀態、查看記錄，以及檢視事件。接著，您會使用 kubectl edit 修正設定錯誤，包括 Service selector 不相符、缺少環境變數，以及無效的 readiness probe 路徑。

本練習執行的工作：

  • 下載專案起始檔案
  • 將資源部署到 Azure（ACR、AKS 叢集）
  • 診斷並解決一些常見問題
  • 清理 Azure 資源

完成本練習大約需要 30 分鐘。

重要：Azure Container Registry task 執行目前不支援 Azure 免費額度。此練習需要 Pay-As-You-Go 或其他付費方案。

開始之前

若要完成本練習，您需要：

  • 具有部署所需 Azure 服務權限的 Azure 訂用帳戶。如果您尚未擁有，可註冊一個。
  • 安裝於其中一個支援平台上的 Visual Studio Code。
  • 最新版的 Azure CLI。
  • Kubernetes 命令列工具 kubectl。
  • 選用：Python 3.12 或更新版本。

下載專案起始檔案並部署 Azure 服務

在本節中，您將下載主控台應用程式的起始檔案，並使用指令碼將所需服務部署到您的 Azure 訂用帳戶。Azure Managed Redis 部署需要 5 到 10 分鐘才能完成。

  1. 開啟瀏覽器並輸入以下 URL 以下載起始檔案。該檔案會儲存在您的預設下載位置。

    https://github.com/MicrosoftLearning/mslearn-azure-ai/raw/main/downloads/python/aks-troubleshoot-python.zip

  2. 將檔案複製或移動到系統中您想要進行專案作業的位置，然後將檔案解壓縮到資料夾中。

  3. 啟動 Visual Studio Code (VS Code)，並在功能表中選取 File > Open Folder...，然後選擇包含專案檔案的資料夾。

  4. 專案同時包含 Bash（azdeploy.sh）和 PowerShell（azdeploy.ps1）的部署指令碼。開啟適合您環境的檔案，並變更指令碼頂端的兩個值以符合您的需求，然後儲存變更。Note: 請勿變更指令碼中的任何其他內容。

    "<your-resource-group-name>" # Resource Group name
"<your-azure-region>" # Azure region for the resources

  5. 在功能表列選取 Terminal > New Terminal，以在 VS Code 中開啟終端機視窗。

  6. 執行以下命令登入您的 Azure 帳戶。依照提示選取本練習要使用的 Azure 帳戶與訂用帳戶。

    az login

  7. 執行以下命令，確保您的訂用帳戶具有安裝 AKS 和 ACR 所需的資源提供者。Microsoft.Compute、Microsoft.Network 與 Microsoft.Storage 提供者是 AKS 的相依項目，Azure 通常會自動註冊它們，但明確註冊可避免新訂用帳戶偶爾發生叢集建立失敗。

    az provider register --namespace Microsoft.ContainerService
az provider register --namespace Microsoft.ContainerRegistry
az provider register --namespace Microsoft.Compute
az provider register --namespace Microsoft.Network
az provider register --namespace Microsoft.Storage

  8. 請確認您位於專案根目錄中，然後在終端機中執行適當的命令以啟動部署指令碼。

    Bash

    bash azdeploy.sh

    PowerShell

    ./azdeploy.ps1

    Note: 如果 PowerShell 因指令碼未經數位簽署而封鎖執行，請在相同的終端機工作階段中執行以下命令，然後再次執行部署指令碼。此命令只會變更目前 PowerShell 處理序的執行原則。

    Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass

將資源部署到 Azure

部署指令碼執行後，請依照下列步驟在 Azure 中建立所需資源。

  1. 模型部署完成後，輸入 1 以啟動 Create Azure Container Registry (ACR)。這會建立儲存 API 容器的資源，之後會從該資源提取映像到 AKS 資源中。

  2. ACR 資源建立完成後，輸入 2 以啟動 Build and push API image to ACR。此選項會使用 ACR tasks 建置映像並將其加入 ACR 存放庫。此作業可能需要 3 到 5 分鐘完成。

  3. 映像建置並推送到 ACR 之後，輸入 3 以啟動 Create AKS cluster 選項。這會建立已設定受控識別的 AKS 資源，並授與該服務從 ACR 資源提取映像的權限。

  4. AKS 叢集部署完成後，輸入 4 以啟動 Get AKS credentials for kubectl 選項。這會使用 az aks get-credentials 命令來擷取認證並設定 kubectl。

  5. 認證設定完成後，輸入 5 以啟動 Deploy applications to AKS 選項。這會將 API 部署到 AKS 叢集。

  6. 應用程式部署完成後，輸入 6 以啟動 Check deployment stats 選項。此選項會回報每個資源是否都已成功部署。

    如果所有服務都傳回 successful 訊息，請輸入 7 以結束部署指令碼。

Note: 請保持終端機開啟，本練習中的所有步驟都會在終端機中執行。

疑難排解部署

部署指令碼在名為 aks-troubleshoot 的 namespace 中建立了所有 Kubernetes 資源。Namespace 是在 Kubernetes 叢集中組織與隔離資源的一種方式。它們可讓您將相關資源分組、套用資源配額，以及管理存取控制。當您未指定 namespace 時，資源會建立在 default namespace 中。在本練習中，所有 kubectl 命令都包含 -n aks-troubleshoot，以鎖定正確的 namespace。

驗證部署之後，您將依序處理三個疑難排解情境。在每個情境中，您都會套用一個 manifest 檔案，該檔案會在部署中引入特定錯誤。接著，您會使用 kubectl 命令診斷問題，並編輯設定加以解決。

驗證部署

在本節中，您將確認設定指令碼部署的應用程式在引入錯誤之前可正常執行。

  1. 執行以下命令，驗證 Pod 正在該 namespace 中執行。命令應傳回一個狀態為 Running 且 READY 欄為 1/1 的 Pod。

    kubectl get pods -n aks-troubleshoot

  2. 執行以下命令，驗證 Service 具有 endpoint。命令應傳回一個列出 IP 位址的 endpoint slice。

    kubectl get endpointslices -l kubernetes.io/service-name=api-service -n aks-troubleshoot

  3. 執行以下命令，使用 port-forward 測試連線能力。此命令會從本機建立一條通往叢集中執行之 Service 的通道，讓您可以透過 http://localhost:8080 存取它。

    kubectl port-forward service/api-service 8080:80 -n aks-troubleshoot

  4. 在功能表列選取 Terminal > New Terminal，以在 VS Code 中開啟第二個終端機視窗。執行以下命令以測試連線。您應該會收到包含 "status": "healthy" 的 JSON 回應。

    # Bash
curl http://localhost:8080/healthz

    # PowerShell
Invoke-RestMethod http://localhost:8080/healthz

  5. 切換回執行 port-forward 的終端機，並輸入 ctrl+c 以結束命令。

您已驗證部署可正常運作，接下來將診斷標籤不相符的問題。

診斷標籤不相符

Service 會根據 label selector 將流量路由到 Pod。當標籤不相符時，Service 就不會有 endpoint，請求也會失敗。API 部署時使用標籤為 app: api 的 Pod，而 Service selector 也對應 app: api。在本節中，您會套用一個 Service 設定，將 selector 變更為 app: api-v2，從而中斷連線。

  1. 執行以下命令，套用會建立標籤不相符錯誤的 Service 設定。

    kubectl apply -f k8s/label-mismatch-service.yaml -n aks-troubleshoot

  2. 執行以下命令，確認 Pod 仍在執行。Pod 會顯示 Running 狀態、1/1 ready，且標籤顯示為 app=api。

    kubectl get pods --show-labels -n aks-troubleshoot

  3. 執行以下命令，檢查 Service endpoint slices。命令應傳回一個在 ENDPOINTS 欄顯示 <unset> 的 endpoint slice，表示沒有 Pod 符合 Service selector。

    kubectl get endpointslices -l kubernetes.io/service-name=api-service -n aks-troubleshoot

  4. 執行以下命令，檢視 Service 詳細資料。請在輸出中尋找 Selector 欄位，現在它會顯示 app=api-v2。

    kubectl describe service api-service -n aks-troubleshoot

    這可確認標籤不相符。Service selector 是 app=api-v2，但 Pod 標籤是 app=api。

  5. 執行以下命令，在編輯器中開啟 Service 設定。

    kubectl edit service api-service -n aks-troubleshoot

    Note: kubectl edit 命令會從叢集擷取即時的資源設定，並在本機文字編輯器中開啟它。當您儲存並關閉編輯器時，kubectl 會自動將變更送回 Kubernetes API server，由它驗證並套用到執行中的叢集。所開啟的編輯器取決於您的環境：

    • Bash: 預設會開啟 vi。按 i 進入 insert mode，完成變更後按 Esc，然後輸入 :wq 並按 Enter 以儲存並離開。輸入 :q! 可在不儲存的情況下離開。
    • PowerShell (Windows): 預設會開啟 Notepad。完成變更後，選取 File > Save（或 Ctrl+S），然後關閉視窗。若未儲存即關閉，則會取消此次編輯。

  6. 在編輯器中，找到 selector 區段，並將 app: api-v2 改為 app: api。儲存變更並關閉編輯器。

  7. 執行以下命令，驗證 endpoint slice 位址已恢復。命令應傳回一個列出 IP 位址的 endpoint slice。

    kubectl get endpointslices -l kubernetes.io/service-name=api-service -n aks-troubleshoot

您已修正標籤不相符的問題，接下來將診斷 CrashLoopBackOff。

診斷 CrashLoopBackOff

當容器無法啟動時，Kubernetes 會反覆重新啟動它，導致狀態變成 CrashLoopBackOff。讀取記錄可了解應用程式為何當機。

  1. 執行以下命令，套用會移除必要 API_KEY 環境變數的部署設定。

    kubectl apply -f k8s/crashloop-deployment.yaml -n aks-troubleshoot

  2. 執行以下命令，觀察 Pod 狀態。稍候片刻後，Pod 會進入 CrashLoopBackOff。輸入 ctrl-c 以結束命令。

    kubectl get pods -n aks-troubleshoot -w

  3. 執行以下命令，檢查 Pod 記錄中的錯誤訊息。您應該會看到指出缺少環境變數的錯誤。

    kubectl logs -l app=api -n aks-troubleshoot

  4. 編輯 Deployment 以加入 API_KEY 環境變數，藉此修正問題。

    kubectl edit deployment api-deployment -n aks-troubleshoot

    在編輯器中，於 spec.template.spec 下找到 containers 區段。找到 name: api 這一行，並在其正下方加入 env 區塊，縮排需與 name 相同：

        name: api
     env:
     - name: API_KEY
       value: "demo-api-key-12345"
     ports:

    儲存變更並關閉編輯器。

  5. 執行以下命令，觀察 Pod 狀態。稍候片刻後，Pod 會進入 Running。輸入 ctrl-c 以結束命令。

    kubectl get pods -n aks-troubleshoot -w

您已解決 CrashLoopBackOff 問題，接下來將診斷 readiness probe 失敗。

診斷 readiness probe 失敗

當 readiness probe 失敗時，Pod 會顯示為 Running，但 0/1 個容器處於 ready。Kubernetes 在通過 readiness 檢查之前，不會將該 Pod 加入 Service endpoint。採用 rolling update 策略時，舊的可用 Pod 會持續提供流量，而新的 Pod 則會卡在 not-ready 狀態。

  1. 執行以下命令，套用會引入 readiness probe 失敗的部署設定。這會為 readiness 檢查套用無效路徑。

    kubectl apply -f k8s/probe-failure-deployment.yaml -n aks-troubleshoot

  2. 執行以下命令，檢查 Pod 狀態。您應該會看到兩個 Pod：新的 Pod 顯示為 Running，但 READY 欄為 0/1；舊的 Pod 則仍維持 1/1 ready。rolling update 會被阻塞，因為新的 Pod 一直無法變成 ready。

    kubectl get pods -n aks-troubleshoot

  3. 執行以下命令，檢查 probe 失敗事件。此命令會傳回所有 Unhealthy 事件，其中包含 readiness 與 liveness probe 失敗。請尋找指出 readiness probe 因 404 狀態碼而失敗的訊息。

    kubectl get events -n aks-troubleshoot --field-selector reason=Unhealthy

  4. 執行以下命令，透過編輯 Deployment 以修正路徑，藉此修復 readiness probe。

    kubectl edit deployment api-deployment -n aks-troubleshoot

    在編輯器中，找到 readinessProbe 區段，並將 path: /invalid-path 改為 path: /healthz。儲存變更並關閉編輯器。

  5. 執行以下命令，驗證新 Pod 已變成 ready，且舊 Pod 已終止。您應該只會看到一個狀態為 Running 且 READY 欄為 1/1 的 Pod。

    kubectl get pods -n aks-troubleshoot

您已診斷並解決 readiness probe 錯誤，接下來將驗證端對端連線能力。

驗證端對端連線能力

完成所有疑難排解情境後，您將確認應用程式已完全正常運作。

  1. 執行以下命令，使用 port-forward 存取 Service。

    kubectl port-forward service/api-service 8080:80 -n aks-troubleshoot

  2. 在功能表列選取 Terminal > New Terminal，以在 VS Code 中開啟第二個終端機視窗。執行以下命令以測試所有端點。

    # Bash
curl http://localhost:8080/healthz
curl http://localhost:8080/readyz
curl http://localhost:8080/api/info

    # PowerShell
Invoke-RestMethod http://localhost:8080/healthz
Invoke-RestMethod http://localhost:8080/readyz
Invoke-RestMethod http://localhost:8080/api/info

  3. 執行以下命令，檢查 Pod 記錄以查看這些請求。

    kubectl logs -l app=api -n aks-troubleshoot

您已驗證應用程式完全正常運作，接下來將清理資源。

清理資源

現在您已完成本練習，應刪除所建立的雲端資源，以避免不必要的資源耗用。

  1. 在 VS Code 終端機中執行以下命令，以刪除資源群組及群組中的所有資源。將 <rg-name> 替換為您稍早在練習中選擇的名稱。此命令會在 Azure 中啟動背景工作，以刪除資源群組。

    az group delete --name <rg-name> --no-wait --yes

CAUTION: 刪除資源群組會刪除其中包含的所有資源。若您為本練習選擇了現有的資源群組，則不屬於本練習範圍的既有資源也會一併被刪除。

疑難排解

如果您在設定本練習時遇到問題，請嘗試下列疑難排解步驟：

使用部署指令碼檢查部署狀態

  • 執行部署指令碼並選取選項 6. Check deployment status，以驗證所有已部署資源的狀態。
  • 此命令會檢查 ACR 佈建狀態、AKS 叢集佈建狀態，以及 Kubernetes 資源可用性。
  • 使用此輸出識別可能造成問題的元件。

ACR 映像提取錯誤

  • 如果 Pod 顯示 ImagePullBackOff 或 ErrImagePull 狀態，請確認 ACR 資源已建立且映像已成功推送。
  • 如有需要，請再次執行部署指令碼中的選項 2. Build and push API image to ACR。
  • 確認 AKS 叢集具有從 ACR 提取映像的權限，可檢查部署指令碼輸出中是否成功指派角色。

kubectl 連線問題

  • 如果 kubectl 命令因連線錯誤而失敗，請執行部署指令碼中的選項 4. Get AKS credentials for kubectl 來重新整理認證。
  • 可透過檢查 Azure 入口網站，或執行 az aks show --resource-group <rg-name> --name <aks-name> --query provisioningState，來驗證 AKS 叢集是否正在執行。

重設練習

  • 如果您需要重新開始疑難排解情境，請執行部署指令碼中的選項 5. Deploy applications to AKS，以重新部署原始可正常運作的設定。
  • 這會重新套用基礎 deployment 與 service 檔案，重設您在練習期間所做的任何變更。
