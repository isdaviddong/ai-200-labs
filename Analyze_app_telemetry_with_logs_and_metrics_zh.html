使用 KQL 查詢記錄

Kusto Query Language (KQL) 是用來分析 Application Insights 中記錄資料的查詢語言。KQL 查詢可讓您篩選、彙總及聯結 requests、dependencies 與 exceptions 等遙測資料表，以診斷應用程式健康狀態與效能。Application Insights 中的 Logs 刀鋒視窗提供互動式查詢編輯器，具備自動完成、視覺化結果與時間範圍控制，使其成為調查遙測的主要工具。搭配透過 Azure CLI 建立的排程查詢警示規則，KQL 可實現主動式監視，當失敗率或延遲超過可接受門檻時通知您的團隊。

在本練習中，您將部署 Application Insights 資源、執行一個使用 OpenTelemetry 產生 requests、dependencies 與 exceptions 範例遙測的 Python 指令碼，然後在 Azure portal 的 Logs 刀鋒視窗中撰寫 KQL 查詢，以調查應用程式健康狀態。您會查詢 requests 資料表來找出失敗、將 exceptions 與 requests 聯結以關聯錯誤、使用百分位數計算分析相依性延遲，並使用 Azure CLI 建立動作群組與記錄搜尋警示規則。

本練習將執行的工作：

  • 下載專案起始檔
  • 建立 Application Insights 資源
  • 執行遙測產生器以建立範例資料
  • 在 Azure portal 中使用 KQL 查詢遙測
  • 使用 Azure CLI 建立動作群組與警示規則

本練習大約需要 20 分鐘完成。

開始之前

若要完成本練習，您需要：

  • Azure 訂用帳戶。如果您還沒有，可以 註冊一個。
  • Visual Studio Code 安裝於其中一個 支援平台。
  • Python 3.12 或更新版本。
  • 最新版的 Azure CLI。

下載專案起始檔並部署 Application Insights

在本節中，您將下載應用程式的起始檔，並使用指令碼將 Application Insights 資源部署到您的訂用帳戶。

  1. 開啟瀏覽器並輸入下列 URL 以下載起始檔。檔案會儲存在您的預設下載位置。

    https://github.com/MicrosoftLearning/mslearn-azure-ai/raw/main/downloads/python/analyze-logs-python.zip

  2. 將檔案複製或移動到您系統中要進行專案作業的位置，然後將檔案解壓縮到資料夾中。

  3. 啟動 Visual Studio Code (VS Code)，然後在功能表中選取 File > Open Folder...，接著選擇包含專案檔案的資料夾。

  4. 此專案同時包含 Bash (azdeploy.sh) 與 PowerShell (azdeploy.ps1) 的部署指令碼。請開啟符合您環境的檔案，修改指令碼頂端的兩個值以符合您的需求，然後儲存變更。注意： 請勿變更指令碼中的其他任何內容。

    "<your-resource-group-name>" # Resource Group name
"<your-azure-region>" # Azure region for the resources

  5. 在功能表列中選取 Terminal > New Terminal，以在 VS Code 中開啟終端機視窗。

  6. 執行下列命令登入您的 Azure 帳戶。依照提示選取本練習要使用的 Azure 帳戶與訂用帳戶。

    az login

  7. 執行下列命令，確認您的訂用帳戶具有本練習所需的資源提供者。

    az provider register --namespace Microsoft.Insights
az provider register --namespace Microsoft.OperationalInsights

  8. 執行下列命令以新增 Application Insights CLI 擴充功能。此擴充功能提供部署指令碼用來建立及管理 Application Insights 資源的命令。

    az extension add --name application-insights

  9. 在終端機中執行適當的命令以啟動指令碼。

    Bash

    bash azdeploy.sh

    PowerShell

    ./azdeploy.ps1

    Note: 如果 PowerShell 因指令碼未經數位簽署而封鎖它，請在相同的終端機工作階段中執行下列命令，然後再次執行部署指令碼。此命令只會變更目前 PowerShell 程序的執行原則。

    Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass

  10. 當指令碼執行時，輸入 1 以啟動 1. Create Application Insights 選項。

    此選項會在資源群組尚不存在時建立它，並建立 Application Insights 資源。

  11. 輸入 2 以執行 2. Assign role 選項。這會將 Monitoring Metrics Publisher 角色指派給您的帳戶，讓應用程式可使用 Microsoft Entra 驗證將遙測發佈到 Application Insights。

  12. 輸入 3 以執行 3. Check deployment status 選項。確認 Application Insights 資源顯示為 Succeeded 且角色已指派後再繼續。如果資源仍在佈建中，請稍候再試。

  13. 輸入 4 以執行 4. Retrieve connection info 選項。這會建立環境變數檔案，其中包含應用程式與 CLI 命令所需的 Application Insights 連接字串、資源群組名稱、Application Insights 名稱與資源 ID。

  14. 輸入 5 以結束部署指令碼。

  15. 執行適當的命令，將前一步建立的檔案中的環境變數載入到您的終端機工作階段中。

    Bash

    source .env

    PowerShell

    . .\.env.ps1

    Note: 請保持終端機開啟。如果您將其關閉並建立新的終端機，則需要再次執行此命令以重新載入環境變數。

產生遙測資料

在本節中，您將設定 Python 環境、執行預先撰寫好的遙測產生器，以在 Application Insights 中建立 requests、dependencies 與 exceptions 的範例資料，然後等待資料送達後再進行查詢。產生器指令碼會使用 OpenTelemetry 建立 Span，這些 Span 會對應到 Application Insights 中的 requests、dependencies 與 exceptions 資料表。

  1. 在 VS Code 終端機中執行下列命令，以切換到 client 目錄。

    cd client

  2. 執行下列命令以建立 Python 環境。

    python -m venv .venv

  3. 執行下列命令以啟用 Python 環境。注意： 在 Linux/macOS 上，請使用 Bash 命令；在 Windows 上，請使用 PowerShell 命令。如果您在 Windows 上使用 Git Bash，請使用 source .venv/Scripts/activate。

    Bash

    source .venv/bin/activate

    PowerShell

    .\.venv\Scripts\Activate.ps1

  4. 執行下列命令以安裝相依性。

    pip install -r requirements.txt

  5. 執行下列命令以啟動遙測產生器。

    python app.py

  6. 此指令碼會產生 15 個 request Span、12 個 dependency Span 與 5 個 exception Span，然後列印摘要。請等待指令碼執行完成。

  7. 再執行兩次指令碼以產生額外的遙測資料。

    python app.py

  8. 等待兩到三分鐘，讓遙測送達 Application Insights。遙測會以批次方式定期傳送，因此資料出現前會有短暫延遲。

在 Azure portal 中查詢遙測

在本節中，您將使用 Application Insights 中的 Logs 刀鋒視窗，對您產生的遙測執行 KQL 查詢。Logs 刀鋒視窗提供具備自動完成、表格式結果與圖表呈現的互動式編輯器。

  1. 前往 Azure portal，並在您先前建立的資源群組中找到 Application Insights 資源。

  2. 在 Application Insights 資源中，於左側導覽的 Monitoring 下選取 Logs。關閉任何出現的查詢範本對話方塊。注意： 請務必在查詢列的下拉選擇器中選取 KQL mode。

查詢失敗的要求

  1. 將下列查詢複製並貼到查詢編輯器中，然後選取 Run。此查詢使用 where 運算子將 requests 資料表篩選為僅包含失敗的要求，接著使用 summarize 依服務名稱與 HTTP 狀態碼分組。count() 彙總會統計各種組合的失敗次數，而 order by 會排序結果，使最常出現的失敗優先顯示。

    requests
| where success == false
| summarize failedCount = count() by cloud_RoleName, resultCode
| order by failedCount desc

  2. 檢閱結果。您應該會看到 api-gateway、doc-processor 與 auth-service 服務的資料列，狀態碼為 500 與 429。failedCount 欄位會顯示每種組合發生了多少次失敗。

查詢要求量與效能

  1. 將下列查詢複製並貼到查詢編輯器中，然後選取 Run。此查詢使用 bin(timestamp, 1h) 將要求分桶為每小時區間，並依服務名稱分組。對於每個區間，它會計算總要求數、使用 avg() 計算平均持續時間，以及使用 percentile() 計算第 95 百分位持續時間。第 95 百分位表示 95% 要求可在其內完成的回應時間，因此對偵測尾端延遲特別有用。

    requests
| summarize requestCount = count(),
    avgDuration = avg(duration),
    p95Duration = percentile(duration, 95)
    by bin(timestamp, 1h), cloud_RoleName
| order by timestamp desc

  2. 檢閱結果。p95Duration 欄位揭示最慢 5% 要求所經歷的回應時間。請比較平均持續時間與第 95 百分位 —— 若差距很大，表示大多數要求很快，但其中一部分有顯著延遲。

將 exceptions 與 requests 聯結

  1. 將下列查詢複製並貼到查詢編輯器中，然後選取 Run。此查詢從 exceptions 資料表開始，並使用 join kind=inner 透過共用的 operation_Id 欄位，將每個例外狀況與觸發它的要求相符。子查詢中的 project 會在聯結前將 name 重新命名為 requestName，以避免欄位名稱衝突，因為兩個資料表都具有 name 欄位。聯結之後，summarize 會依要求名稱、例外狀況類型與服務分組結果，統計每種組合產生了多少例外狀況。take 15 運算子則將輸出限制為前 15 列。

    exceptions
| join kind=inner (requests | project requestName = name, operation_Id, cloud_RoleName) on operation_Id
| summarize exceptionCount = count()
    by requestName, exceptionType = type, cloud_RoleName
| order by exceptionCount desc
| take 15

  2. 檢閱結果。每一列都顯示要求名稱、例外狀況類型與服務的組合。此檢視可協助您識別哪些作業產生最多錯誤，以及涉及哪些例外狀況類型。

分析相依性延遲

  1. 將下列查詢複製並貼到查詢編輯器中，然後選取 Run。此查詢會讀取 dependencies 資料表，該表記錄了應用程式對資料庫、API 與其他服務發出的輸出呼叫。它會依 target（相依性端點）與 type（例如 HTTP 或 SQL）分組，然後計算呼叫次數、平均持續時間，以及第 50、95 與 99 百分位延遲。比較多個百分位數可揭示慢回應是零星離群值，還是更廣泛的模式。

    dependencies
| summarize callCount = count(),
    avgDuration = avg(duration),
    p50 = percentile(duration, 50),
    p95 = percentile(duration, 95),
    p99 = percentile(duration, 99)
    by target, type
| order by p95 desc

  2. 檢閱結果。若 p50 與 p95 值之間差距很大，表示效能不一致：大多數呼叫很快，但有相當比例耗時明顯更久。

建立動作群組與警示規則

在本節中，您將使用 Azure CLI 建立一個動作群組，以及一個在失敗率超過門檻時進行偵測的記錄搜尋警示規則。

  1. 在 VS Code 終端機中執行下列命令，以建立含有電子郵件通知的動作群組。當您執行部署指令碼中的 Retrieve connection info 選項時，ALERT_EMAIL 變數已從您的 Azure 帳戶填入。

    Bash

    az monitor action-group create \
    --resource-group $RESOURCE_GROUP \
    --name pipeline-alerts-ag \
    --short-name PipeAlert \
    --action email oncall-email $ALERT_EMAIL

    PowerShell

    az monitor action-group create `
    --resource-group $env:RESOURCE_GROUP `
    --name pipeline-alerts-ag `
    --short-name PipeAlert `
    --action email oncall-email $env:ALERT_EMAIL

    此命令會建立名為 pipeline-alerts-ag 的動作群組，並在觸發時傳送電子郵件通知。

  2. 執行下列命令以建立記錄搜尋警示規則，監視 Application Insights 資源中的失敗要求。此規則會在五分鐘視窗內評估查詢，若失敗要求超過十筆就會觸發。

    Bash

    az monitor scheduled-query create \
    --resource-group $RESOURCE_GROUP \
    --name high-failure-rate-alert \
    --scopes $APPINSIGHTS_RESOURCE_ID \
    --condition "count 'FailedRequests' > 10" \
    --condition-query FailedRequests="requests | where success == false" \
    --evaluation-frequency 5m \
    --window-size 5m \
    --severity 1 \
    --description "Alert when more than 10 requests fail in a 5-minute window"

    PowerShell

    az monitor scheduled-query create `
    --resource-group $env:RESOURCE_GROUP `
    --name high-failure-rate-alert `
    --scopes $env:APPINSIGHTS_RESOURCE_ID `
    --condition "count 'FailedRequests' > 10" `
    --condition-query FailedRequests="requests | where success == false" `
    --evaluation-frequency 5m `
    --window-size 5m `
    --severity 1 `
    --description "Alert when more than 10 requests fail in a 5-minute window"

    嚴重性設為 1（Error），表示問題重大但尚未達到中斷層級。當警示觸發時，它會啟動動作群組並傳送電子郵件。

  3. 執行下列命令，以確認警示規則已建立且已啟用。

    Bash

    az monitor scheduled-query show \
    --resource-group $RESOURCE_GROUP \
    --name high-failure-rate-alert \
    --query "{Name:name, Severity:severity, Enabled:enabled, Window:windowSize, Frequency:evaluationFrequency, Threshold:condition.failingPeriods}" \
    --output table

    PowerShell

    az monitor scheduled-query show `
    --resource-group $env:RESOURCE_GROUP `
    --name high-failure-rate-alert `
    --query "{Name:name, Severity:severity, Enabled:enabled, Window:windowSize, Frequency:evaluationFrequency, Threshold:condition.failingPeriods}" `
    --output table

    確認 Enabled 欄位顯示 True，且嚴重性、視窗與頻率值與您前一步指定的內容相符。

  4. 執行下列命令，以列出資源群組中的所有排程查詢警示規則。

    Bash

    az monitor scheduled-query list \
    --resource-group $RESOURCE_GROUP \
    --output table

    PowerShell

    az monitor scheduled-query list `
    --resource-group $env:RESOURCE_GROUP `
    --output table

    您應該會看到一條名為 high-failure-rate-alert 的規則。由於您已經產生含有失敗要求的遙測，此規則會立即在其五分鐘週期上開始評估。如果視窗中的失敗要求數超過門檻，Azure Monitor 就會觸發警示，而動作群組會將電子郵件通知傳送到您指定的電子郵件地址。

清除資源

完成本練習後，您應刪除所建立的雲端資源，以避免不必要的資源使用。

  1. 在 VS Code 終端機中執行下列命令，以刪除資源群組及其中所有資源。請將 <rg-name> 取代為您在本練習稍早選擇的名稱。此命令會在 Azure 中啟動背景工作來刪除資源群組。

    az group delete --name <rg-name> --no-wait --yes

注意： 刪除資源群組會刪除其中包含的所有資源。如果您在本練習中選擇的是現有資源群組，則本練習範圍外的任何既有資源也會一併刪除。

疑難排解

如果您在完成本練習時遇到問題，請嘗試以下疑難排解步驟：

驗證 Application Insights 部署

  • 前往 Azure portal 並找到您的資源群組。
  • 確認 Application Insights 資源的 Provisioning State 顯示為 Succeeded。

檢查連接字串

  • 執行部署指令碼中的 Check deployment status 選項，以確認資源已成功建立。
  • 確認 .env 檔案包含 APPLICATIONINSIGHTS_CONNECTION_STRING 值。
  • 如果缺少連接字串，請再次執行 Retrieve connection info 選項。

驗證環境變數

  • 檢查專案根目錄中是否存在 .env 檔案，且其中包含 APPLICATIONINSIGHTS_CONNECTION_STRING、RESOURCE_GROUP、APPINSIGHTS_NAME、APPINSIGHTS_RESOURCE_ID 與 ALERT_EMAIL 值。
  • 確認您已執行 source .env（Bash）或 . ..env.ps1（PowerShell），將環境變數載入到終端機工作階段中。
  • 如果變數是空的，請重新執行 source .env（Bash）或 . ..env.ps1（PowerShell）。

檢查驗證

  • 執行 az account show，確認您已登入 Azure CLI。
  • 請透過檢查 Azure portal 中的角色指派，或再次執行部署指令碼中的角色指派選項，確認 Monitoring Metrics Publisher 角色已指派給您的帳戶。

檢查 Python 環境與相依性

  • 在執行指令碼前，請確認虛擬環境已啟用。
  • 執行 pip list，確認 requirements.txt 中的所有套件都已成功安裝。
  • 如果尚未安裝 azure-monitor-opentelemetry，請再次執行 pip install -r requirements.txt。

遙測未出現在查詢結果中

  • 指令碼送出遙測後，可能需要兩到五分鐘才會出現。請等待，然後在 Logs 刀鋒視窗中再次選取 Run。
  • 請將連接字串與 Azure portal 中 Application Insights 資源 Overview 頁面顯示的值進行比對，以確認其正確。
  • 檢查 VS Code 終端機輸出中是否有任何與遙測匯出相關的錯誤。
  • 如果查詢未傳回結果，請使用 Logs 刀鋒視窗中的時間選擇器擴大時間範圍（例如，將 Last 1 hour 改為 Last 4 hours）。

警示規則建立失敗

  • 請執行 az extension add --name scheduled-query，確認已安裝 scheduled-query 擴充功能。
  • 請執行 echo $APPINSIGHTS_RESOURCE_ID，確認 APPINSIGHTS_RESOURCE_ID 環境變數包含有效的資源 ID。
  • 如果命令需要預覽版擴充功能，請先執行 az config set extension.dynamic_install_allow_preview=true。
