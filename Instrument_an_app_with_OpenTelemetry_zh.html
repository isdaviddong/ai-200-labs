使用 OpenTelemetry 為應用程式加入檢測

OpenTelemetry 是一個開放原始碼可觀測性架構，提供標準化方式來從應用程式收集追蹤、計量與記錄。Azure Monitor OpenTelemetry Distro 將 OpenTelemetry SDK 與 Azure Monitor 匯出工具整合在一起，讓 Python 應用程式只需最少設定即可將遙測傳送到 Application Insights。自訂 Span 可讓您追蹤特定於應用程式的作業，並加入屬性，以商務脈絡豐富追蹤資料。

在本練習中，您將部署 Application Insights 資源，並建置一個示範文件處理管線 OpenTelemetry 檢測的 Python Flask Web 應用程式。您會設定 Azure Monitor OpenTelemetry Distro、為每個管線階段建立自訂父 Span 與子 Span、加入 Span 屬性以擷取文件中繼資料，並使用 Azure portal 中的交易搜尋與記錄查詢來驗證遙測並診斷模擬的延遲瓶頸。

本練習將執行的工作：

  • 下載專案起始檔
  • 建立 Application Insights 資源
  • 在起始檔中加入程式碼以完成應用程式
  • 執行應用程式並在 Application Insights 中診斷效能問題

本練習大約需要 25 分鐘完成。

開始之前

若要完成本練習，您需要：

  • Azure 訂用帳戶。如果您還沒有，可以 註冊一個。
  • Visual Studio Code 安裝於其中一個 支援平台。
  • Python 3.12 或更新版本。
  • 最新版的 Azure CLI。

加入設定遙測的程式碼

在本節中，您將加入程式碼來設定 Azure Monitor OpenTelemetry Distro，讓應用程式將追蹤匯出到 Application Insights。此函式會從環境變數讀取連接字串、建立用於 Microsoft Entra 驗證的 DefaultAzureCredential，並設定 Azure Monitor 匯出工具。此認證會排除受控識別提供者，因為應用程式是在本機執行；若沒有此設定，認證鏈就會在每次匯出遙測時嘗試連線到 Azure Instance Metadata Service，而這些失敗的 HTTP 呼叫會在 Application Map 中形成雜訊。

此函式會呼叫 Azure Monitor OpenTelemetry Distro 套件中的 configure_azure_monitor()。這個單一呼叫會使用 Azure Monitor 追蹤匯出工具設定 OpenTelemetry SDK，並為 Flask 要求設定自動檢測。credential 參數會啟用以 Entra 為基礎的驗證，讓應用程式使用 Monitoring Metrics Publisher 角色來發佈遙測，而不是使用檢測金鑰。部署指令碼在 .env 檔案中設定的 OTEL_SERVICE_NAME 環境變數，會控制出現在 Application Map 上的 cloud.role.name。

  1. 找出 # BEGIN CONFIGURE TELEMETRY FUNCTION 註解，並在該註解下方加入下列程式碼。請務必確認程式碼對齊正確。

    def configure_telemetry():
    """Configure the Azure Monitor OpenTelemetry Distro."""
    connection_string = os.environ.get("APPLICATIONINSIGHTS_CONNECTION_STRING")

    if not connection_string:
        raise ValueError(
            "APPLICATIONINSIGHTS_CONNECTION_STRING environment variable must be set"
        )

    from azure.monitor.opentelemetry import configure_azure_monitor

    credential = DefaultAzureCredential(
        exclude_managed_identity_credential=True
    )

    configure_azure_monitor(
        connection_string=connection_string,
        credential=credential,
    )

  2. 花幾分鐘檢視這段程式碼。

加入處理文件的程式碼

在本節中，您將加入程式碼，為批次文件處理作業建立父 Span。此函式會迴圈處理可設定數量的文件，並為每份文件呼叫三個子 Span 函式：validate、enrich 與 store。

此函式使用 start_as_current_span() 建立名為 "process-documents" 的父 Span，以包覆整個批次。每個子函式都會建立自己的 Span，並自動成為目前 Span 的子 Span，從而建立階層式追蹤樹。Span 屬性會記錄批次大小以及成功處理的文件數量。

  1. 找出 # BEGIN PROCESS DOCUMENTS FUNCTION 註解，並在該註解下方加入下列程式碼。請務必確認程式碼對齊正確。

    def process_documents(doc_count):
    """Process a batch of documents through the pipeline with tracing."""
    tracer = get_tracer()
    results = []

    with tracer.start_as_current_span("process-documents") as parent_span:
        parent_span.set_attribute("document.count", doc_count)
        parent_span.set_attribute("pipeline.name", "document-processing")

        for i in range(1, doc_count + 1):
            doc_id = f"DOC-{i:04d}"

            validate_result = validate_document(doc_id)
            enrich_result = enrich_document(doc_id)
            store_result = store_document(doc_id)

            results.append({
                "doc_id": doc_id,
                "validate": validate_result,
                "enrich": enrich_result,
                "store": store_result
            })

        parent_span.set_attribute("document.processed", len(results))

    return results

  2. 儲存您的變更，並花幾分鐘檢視程式碼。

加入追蹤管線階段的程式碼

在本節中，您將加入三個函式，每個函式都會為文件管線的一個階段建立子 Span：validate、enrich 與 store。三者都遵循相同模式：呼叫 start_as_current_span() 建立會自動成為作用中父 Span 子項的 Span，接著呼叫 set_attribute() 附加可搜尋的中繼資料，再呼叫 set_status() 標記結果。

enrich_document 函式也包含刻意設計的延遲問題。文件 DOC-0003 與 DOC-0005 會經歷 1.5 到 3 秒的延遲，以模擬外部服務瓶頸。enrichment.slow 屬性會標示受影響的 Span，讓您可以在 Application Insights 中篩選它們。當您稍後檢視端對端交易檢視時，這些 Span 會明顯顯示為管線延遲的來源。

  1. 找出 # BEGIN PIPELINE STAGE FUNCTIONS 註解，並在該註解下方加入下列程式碼。請務必確認程式碼對齊正確。

    def validate_document(doc_id):
    """Validate a document and record a traced span."""
    tracer = get_tracer()

    with tracer.start_as_current_span("validate-document") as span:
        span.set_attribute("document.id", doc_id)
        span.set_attribute("document.stage", "validate")

        # Simulate validation work
        time.sleep(random.uniform(0.05, 0.15))
        is_valid = True

        span.set_attribute("document.valid", is_valid)
        span.set_status(StatusCode.OK)

    return {"status": "valid", "duration_ms": round(random.uniform(50, 150))}


def enrich_document(doc_id):
    """Enrich a document with metadata and record a traced span."""
    tracer = get_tracer()

    with tracer.start_as_current_span("enrich-document") as span:
        span.set_attribute("document.id", doc_id)
        span.set_attribute("document.stage", "enrich")

        # Simulated latency issue: documents DOC-0003 and DOC-0005
        # experience high latency during enrichment, representing
        # a bottleneck for the student to diagnose in Application Insights
        if doc_id in ("DOC-0003", "DOC-0005"):
            delay = random.uniform(1.5, 3.0)
            span.set_attribute("enrichment.slow", True)
        else:
            delay = random.uniform(0.05, 0.2)
            span.set_attribute("enrichment.slow", False)

        time.sleep(delay)
        span.set_attribute("enrichment.duration_s", round(delay, 3))
        span.set_status(StatusCode.OK)

    return {
        "status": "enriched",
        "duration_ms": round(delay * 1000),
        "slow": doc_id in ("DOC-0003", "DOC-0005")
    }


def store_document(doc_id):
    """Store a document and record a traced span."""
    tracer = get_tracer()

    with tracer.start_as_current_span("store-document") as span:
        span.set_attribute("document.id", doc_id)
        span.set_attribute("document.stage", "store")
        span.set_attribute("storage.type", "blob")

        # Simulate storage write
        time.sleep(random.uniform(0.05, 0.2))

        span.set_status(StatusCode.OK)

    return {"status": "stored", "duration_ms": round(random.uniform(50, 200))}

  2. 儲存您的變更，並花幾分鐘檢視程式碼。

執行應用程式

在本節中，您將執行完成的 Flask 應用程式以產生遙測，然後在 Azure portal 中使用 Transaction search 與記錄查詢來驗證 Span，並診斷模擬的效能瓶頸。

  1. 在終端機中執行下列命令以啟動應用程式。如有需要，請先參考本練習前面的命令來啟用環境，再執行此命令。如果您已離開 client 目錄，請先執行 cd client。

    python app.py

  2. 開啟瀏覽器並前往 http://localhost:5000 以存取應用程式。

  3. 在左側面板中選取 Check Telemetry Status。確認遙測狀態顯示為 active，且資源屬性包含值為 document-pipeline-app 的 service.name。這可確認 Azure Monitor OpenTelemetry Distro 已設定並正在匯出遙測。

  4. 在左側面板中選取 Process Documents。這會讓五份文件通過管線處理，並以表格顯示結果。請注意，文件 DOC-0003 與 DOC-0005 會顯示明顯較高的 enrichment 持續時間與 SLOW 標籤，而其他文件則會快速完成。

  5. 再選取兩次 Process Documents 以產生額外的遙測資料。每次執行都會建立含有父 Span 與子 Span 的新追蹤。

  6. 等待兩到三分鐘，讓遙測送達 Application Insights。遙測會以批次方式定期傳送，因此資料出現在 portal 前會有短暫延遲。

  7. 前往 Azure portal，並在您先前建立的資源群組中找到 Application Insights 資源。

檢視端對端交易

  1. 在 Application Insights 資源中，於左側導覽的 Investigate 下選取 Search。此檢視會列出所有傳入要求及其相關遙測。

  2. 在結果清單中，找到其中一筆 POST /process-documents 項目並選取它。端對端交易檢視會開啟，顯示完整的 Span 階層：根 HTTP 要求 Span、"process-documents" 父 Span，以及每個管線階段（validate、enrich、store）的子 Span。

  3. 在交易時間軸中，找出持續時間為 1.5 秒以上的 "enrich-document" Span。這些就是文件 DOC-0003 與 DOC-0005 顯示模擬延遲的 Span。選取其中一個 Span 以檢視其屬性，包括 document.id、document.stage 與 enrichment.slow = True。

使用 KQL 查詢遙測

  1. 在 Application Insights 資源中，於左側導覽的 Monitoring 下選取 Logs。關閉任何出現的查詢範本對話方塊。注意： 請務必在查詢列的下拉選擇器中選取 KQL mode。

  2. 將下列查詢複製並貼到查詢編輯器中，然後選取 Run。此查詢會擷取程式碼建立的自訂 Span，以及您設定的 Span 屬性。

    dependencies
| where timestamp > ago(1h)
| project timestamp, name, duration,
    documentId = customDimensions["document.id"],
    stage = customDimensions["document.stage"],
    slow = customDimensions["enrichment.slow"]
| order by timestamp desc

  3. 檢閱結果。您應該會看到每個管線階段的資料列 —— validate-document、enrich-document 與 store-document —— 並包含您在程式碼中加入的 documentId 與 stage 屬性。slow 欄位會在 DOC-0003 與 DOC-0005 的資料列中顯示 True。

  4. 複製並貼上下列查詢，以比較慢速與快速 enrichment Span 的平均持續時間。

    dependencies
| where timestamp > ago(1h) and name == "enrich-document"
| extend slow = tostring(customDimensions["enrichment.slow"])
| summarize avgDuration = round(avg(duration), 0) by slow

  5. 檢閱結果。True 資料列會顯示 1,500 毫秒以上的平均持續時間，而 False 資料列則顯示低於 200 毫秒的平均值。這確認 enrichment 階段就是瓶頸，且 Span 屬性能清楚識別哪些文件受到影響。

疑難排解

如果您在完成本練習時遇到問題，請嘗試以下疑難排解步驟：

驗證 Application Insights 部署

  • 前往 Azure portal 並找到您的資源群組。
  • 確認 Application Insights 資源的 Provisioning State 顯示為 Succeeded。

檢查連接字串

  • 執行部署指令碼中的 Check deployment status 選項，以確認資源已成功建立。
  • 確認 .env 檔案包含 APPLICATIONINSIGHTS_CONNECTION_STRING 值。
  • 如果缺少連接字串，請再次執行 Retrieve connection info 選項。

檢查程式碼完整性與縮排

  • 請確認所有程式碼區塊都已加入 telemetry_functions.py 中適當的區段，且位於正確的 BEGIN/END 註解標記之間。
  • 確認 Python 縮排一致（使用空格，不要使用 Tab），且所有程式碼在函式內都正確對齊。
  • 確認沒有在指定區段之外誤刪或修改任何程式碼。

驗證環境變數

  • 檢查專案根目錄中是否存在 .env 檔案，且其中包含 APPLICATIONINSIGHTS_CONNECTION_STRING 值。
  • 確認您已執行 source .env（Bash）或 . ..env.ps1（PowerShell），將環境變數載入到終端機工作階段中。
  • 如果變數是空的，請重新執行 source .env（Bash）或 . ..env.ps1（PowerShell）。

檢查驗證

  • 執行 az account show，確認您已登入 Azure CLI。
  • 請透過檢查 Azure portal 中的角色指派，或再次執行部署指令碼中的角色指派選項，確認 Monitoring Metrics Publisher 角色已指派給您的帳戶。

檢查 Python 環境與相依性

  • 在執行應用程式前，請確認虛擬環境已啟用。
  • 執行 pip list，確認 requirements.txt 中的所有套件都已成功安裝。
  • 如果尚未安裝 azure-monitor-opentelemetry，請再次執行 pip install -r requirements.txt。

遙測未出現在 Application Insights 中

  • 應用程式送出遙測後，可能需要兩到五分鐘才會出現在 portal 中。請等待並重新整理查詢結果。
  • 請將連接字串與 Azure portal 中 Application Insights 資源 Overview 頁面顯示的值進行比對，以確認其正確。
  • 檢查 VS Code 終端機輸出中是否有任何與遙測匯出相關的錯誤。
  • 如果 KQL 查詢未傳回結果，請擴大 where 子句中的時間範圍（例如，將 ago(1h) 改為 ago(4h)）。
