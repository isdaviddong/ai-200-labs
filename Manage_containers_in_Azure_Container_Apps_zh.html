診斷並修正失敗的部署

在本練習中，您會針對失敗的容器應用程式進行疑難排解，並套用有針對性的修正。您會使用修訂狀態、記錄及 Azure CLI 來隔離部署問題。此工作流程在 AI 解決方案中很常見，因為當您更新模型與相依性時，啟動行為會經常變化。

本練習中將完成下列工作：

  • 將模擬 AI 文件處理 API 部署為容器應用程式
  • 製造並診斷遺失環境變數的錯誤
  • 製造並診斷輸入設定問題
  • 查詢 Log Analytics 以取得歷史疑難排解資料

本練習約需 30 分鐘完成。

重要： Azure Container Registry task runs 目前不支援 Azure 免費額度。本練習需要 Pay-As-You-Go 或其他付費方案。

開始之前

若要完成本練習，您需要：

  • 具備部署所需 Azure 服務權限的 Azure 訂閱。如果您還沒有，可以 註冊一個。
  • Visual Studio Code ，可在下列其中一個 支援的平台上執行。
  • 最新版的 Azure CLI。
  • 選用： Python 3.12 或更新版本。

下載專案起始檔並部署 Azure 服務

在本節中，您會下載專案起始檔，並使用指令碼將所需服務部署到 Azure 訂閱。Azure Container Registry 與 Container Apps 環境的部署需要幾分鐘才能完成。

  1. 開啟瀏覽器並輸入下列 URL 以下載起始檔。檔案會儲存在您的預設下載位置。

    https://github.com/MicrosoftLearning/mslearn-azure-ai/raw/main/downloads/python/aca-manage-python.zip

  2. 將檔案複製或移動到您希望進行專案作業的系統位置，然後將其解壓縮到資料夾中。

  3. 啟動 Visual Studio Code (VS Code)，並選取 File > Open Folder... ，然後選擇包含專案檔案的資料夾。

  4. 此專案同時包含 Bash（azdeploy.sh）與 PowerShell（azdeploy.ps1）。開啟符合您環境的檔案，將指令碼頂端的兩個值修改為符合您的需求，然後儲存變更。 注意： 不要變更指令碼中的其他任何內容。

    "<your-resource-group-name>" # Resource Group name
"<your-azure-region>" # Azure region for the resources

  5. 在功能表列中選取 Terminal > New Terminal ，在 VS Code 中開啟終端機視窗。

  6. 執行下列命令登入您的 Azure 帳戶。依照提示選取本練習要使用的 Azure 帳戶與訂閱。

    az login

  7. 執行下列命令以確認您已安裝 Azure CLI 的 containerapp 延伸模組。

    az extension add --name containerapp

  8. 執行下列命令，以確保您的訂閱具有本練習所需的資源提供者。

    az provider register --namespace Microsoft.App
az provider register --namespace Microsoft.OperationalInsights
az provider register --namespace Microsoft.ContainerRegistry

在 Azure 中建立資源

在本節中，您會執行部署指令碼，將所需服務部署到您的 Azure 訂閱。

  1. 請確認您位於專案根目錄，並在終端機中執行對應命令來啟動部署指令碼。部署指令碼會部署 ACR，並建立本練習所需環境變數的檔案。

    Bash

    bash azdeploy.sh

    PowerShell

    ./azdeploy.ps1

    注意： 如果 PowerShell 因指令碼未經數位簽署而封鎖它，請在同一個終端機工作階段中執行下列命令，然後再次執行部署指令碼。此命令只會變更目前 PowerShell 處理序的執行原則。

    Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass

  2. 當指令碼執行時，輸入 1 以啟動 Create Azure Container Registry and build container image 選項。此選項會建立 ACR 服務，並使用 ACR Tasks 建置與推送映像到登錄。

  3. 前一個作業完成後，輸入 2 以啟動 Create Container Apps environment 選項。部署容器之前必須先建立環境。

  4. 前一個作業完成後，輸入 3 以啟動 Deploy the container app and configure secrets option.

    注意： 容器應用程式建立後，系統會產生一個包含環境變數的檔案。您會在整個練習中使用這些變數。

  5. 前一個作業完成後，輸入 5 以結束部署指令碼。

  6. 執行對應命令，從前一步建立的檔案中將環境變數載入目前的終端機工作階段。

    Bash

    source .env

    PowerShell

    . .\.env.ps1

    注意： 請保持終端機開啟。如果您將它關閉並建立新的終端機，可能需要再次執行命令來建立環境變數。

  7. 執行下列命令以取得應用程式的 FQDN，並將結果儲存到變數中。

    Bash

    FQDN=$(az containerapp show -n $CONTAINER_APP_NAME -g $RESOURCE_GROUP \
    --query properties.configuration.ingress.fqdn -o tsv)

echo "$FQDN"

    PowerShell

    $FQDN = az containerapp show -n $env:CONTAINER_APP_NAME -g $env:RESOURCE_GROUP `
    --query properties.configuration.ingress.fqdn -o tsv

Write-Output $FQDN

  8. 執行下列命令以呼叫預設端點，確認應用程式正在執行。此命令應回傳一些 JSON。請查看 model.name 欄位，其值應設為 gpt-4o-mini.

    Bash

    curl -s "https://$FQDN/"

    PowerShell

    Invoke-RestMethod -Uri "https://$FQDN/"

診斷遺失的環境變數

當容器應用程式依賴尚未設定的環境變數時，應用程式可能無法啟動或出現非預期行為。在本節中，您會移除必要的環境變數並觀察症狀。

  1. 執行下列命令以更新容器應用程式，移除 MODEL_NAME 環境變數。

    Bash

    az containerapp update -n $CONTAINER_APP_NAME -g $RESOURCE_GROUP \
    --remove-env-vars MODEL_NAME

    PowerShell

    az containerapp update -n $env:CONTAINER_APP_NAME -g $env:RESOURCE_GROUP `
    --remove-env-vars MODEL_NAME

  2. 執行下列命令以列出修訂，確認已建立新的修訂。請尋找尾碼數字較高的新修訂（例如 ai-api--0000002）以及 TrafficWeight 為 100，表示它目前正在接收所有流量。

    Bash

    az containerapp revision list -n $CONTAINER_APP_NAME -g $RESOURCE_GROUP -o table

    PowerShell

    az containerapp revision list -n $env:CONTAINER_APP_NAME -g $env:RESOURCE_GROUP -o table

  3. 執行下列命令以檢查根端點，從 API 使用者的角度觀察症狀。 model.name 欄位現在會顯示預設值 not-configured ，而不是已設定的值。

    Bash

    curl -s "https://$FQDN/" | jq .model

    PowerShell

    (Invoke-RestMethod -Uri "https://$FQDN/").model

  4. 執行下列命令以檢視容器應用程式設定，診斷根本原因。請確認 MODEL_NAME 環境變數已遺失。

    Bash

    az containerapp show -n $CONTAINER_APP_NAME -g $RESOURCE_GROUP \
    --query "properties.template.containers[0].env" -o table

    PowerShell

    az containerapp show -n $env:CONTAINER_APP_NAME -g $env:RESOURCE_GROUP `
    --query "properties.template.containers[0].env" -o table

  5. 執行下列命令，將 MODEL_NAME 環境變數加回來以修正問題。

    Bash

    az containerapp update -n $CONTAINER_APP_NAME -g $RESOURCE_GROUP \
    --set-env-vars MODEL_NAME=$MODEL_NAME

    PowerShell

    az containerapp update -n $env:CONTAINER_APP_NAME -g $env:RESOURCE_GROUP `
    --set-env-vars MODEL_NAME=$env:MODEL_NAME

  6. 執行下列命令，再次檢查根端點以驗證修正結果。這可確認應用程式現在從 API 使用者的角度已能正確運作。回應現在應顯示已設定的模型名稱。

    Bash

    curl -s "https://$FQDN/" | jq .model

    PowerShell

    (Invoke-RestMethod -Uri "https://$FQDN/").model

您已診斷並修正遺失環境變數的問題。接下來，您將診斷輸入設定問題。

診斷輸入設定問題

Container Apps 會使用 target-port 設定將流量路由到您的容器。如果該連接埠與應用程式正在接聽的連接埠不符，要求就會失敗。在本節中，您會製造連接埠不相符的情況。

  1. 執行下列命令以更新容器應用程式，使其使用錯誤的目標連接埠。

    Bash

    az containerapp ingress update -n $CONTAINER_APP_NAME -g $RESOURCE_GROUP \
    --target-port 3000

    PowerShell

    az containerapp ingress update -n $env:CONTAINER_APP_NAME -g $env:RESOURCE_GROUP `
    --target-port 3000

  2. 執行下列命令，嘗試存取健康狀態端點，從 API 使用者的角度觀察症狀。

    Bash

    curl -s "https://$FQDN/health"

    PowerShell

    Invoke-RestMethod -Uri "https://$FQDN/health"

    此要求會失敗或逾時，因為 Container Apps 將流量路由到連接埠 3000，但應用程式實際上正在接聽連接埠 8000。

  3. 執行下列命令以檢查目前的輸入設定，診斷根本原因。請注意 targetPort 已設為 3000。

    Bash

    az containerapp show -n $CONTAINER_APP_NAME -g $RESOURCE_GROUP \
    --query "properties.configuration.ingress" -o yaml

    PowerShell

    az containerapp show -n $env:CONTAINER_APP_NAME -g $env:RESOURCE_GROUP `
    --query "properties.configuration.ingress" -o yaml

  4. 執行下列命令以檢查容器記錄，確認應用程式是否正在執行。您應該會看到 gunicorn 啟動訊息，指出應用程式正在接聽連接埠 8000，藉此確認不相符的情況。

    Bash

    az containerapp logs show -n $CONTAINER_APP_NAME -g $RESOURCE_GROUP

    PowerShell

    az containerapp logs show -n $env:CONTAINER_APP_NAME -g $env:RESOURCE_GROUP

  5. 執行下列命令，將正確的目標連接埠設回去以修正輸入設定。

    Bash

    az containerapp ingress update -n $CONTAINER_APP_NAME -g $RESOURCE_GROUP \
    --target-port 8000

    PowerShell

    az containerapp ingress update -n $env:CONTAINER_APP_NAME -g $env:RESOURCE_GROUP `
    --target-port 8000

  6. 執行下列命令，透過呼叫健康狀態端點來驗證修正結果。這可確認應用程式從 API 使用者的角度已可存取。您應該會看到 {"status":"healthy"}.

    Bash

    curl -s "https://$FQDN/health"

    PowerShell

    Invoke-RestMethod -Uri "https://$FQDN/health"

您已診斷並修正輸入設定問題。接下來，您將學習如何查詢歷史記錄。

查詢 Log Analytics 以進行歷史疑難排解

由 az containerapp logs show 顯示的主控台記錄僅限近期資料。若要進行歷史疑難排解，記錄會保留在與您的 Container Apps 環境關聯的 Log Analytics 工作區中。

  1. 執行下列命令，以從 Container Apps 環境取得 Log Analytics 工作區 ID。

    Bash

    WORKSPACE_ID=$(az containerapp env show -n $ACA_ENVIRONMENT -g $RESOURCE_GROUP \
    --query properties.appLogsConfiguration.logAnalyticsConfiguration.customerId -o tsv)

echo "Workspace ID: $WORKSPACE_ID"

    PowerShell

    $WORKSPACE_ID = az containerapp env show -n $env:ACA_ENVIRONMENT -g $env:RESOURCE_GROUP `
    --query properties.appLogsConfiguration.logAnalyticsConfiguration.customerId -o tsv

Write-Output "Workspace ID: $WORKSPACE_ID"

  2. 執行下列命令以查詢容器應用程式的主控台記錄。這會回傳最後 20 筆記錄項目，顯示時間戳記與訊息。

    Bash

    az monitor log-analytics query -w $WORKSPACE_ID \
    --analytics-query "ContainerAppConsoleLogs_CL | where ContainerAppName_s == '$CONTAINER_APP_NAME' | project TimeGenerated, Log_s | order by TimeGenerated desc | take 20" \
    -o table

    PowerShell

    az monitor log-analytics query -w $WORKSPACE_ID `
    --analytics-query "ContainerAppConsoleLogs_CL | where ContainerAppName_s == '$env:CONTAINER_APP_NAME' | project TimeGenerated, Log_s | order by TimeGenerated desc | take 20" `
    -o table

    [!NOTE] 事件發生後，Log Analytics 資料可能需要幾分鐘才會出現。如果您沒有看到最近的記錄，請稍候幾分鐘後再試一次。

  3. 執行下列命令以專門查詢錯誤層級的記錄。

    Bash

    az monitor log-analytics query -w $WORKSPACE_ID \
    --analytics-query "ContainerAppConsoleLogs_CL | where ContainerAppName_s == '$CONTAINER_APP_NAME' and Log_s contains 'error' | order by TimeGenerated desc | take 20" \
    -o table

    PowerShell

    az monitor log-analytics query -w $WORKSPACE_ID `
    --analytics-query "ContainerAppConsoleLogs_CL | where ContainerAppName_s == '$env:CONTAINER_APP_NAME' and Log_s contains 'error' | order by TimeGenerated desc | take 20" `
    -o table

這些查詢可協助您調查過去發生的問題，即使容器已重新啟動或修訂已變更也一樣。

清除資源

清理可避免持續產生成本。請刪除資源群組，這將一併刪除 Container Apps 環境、容器應用程式與登錄。

az group delete --name $RESOURCE_GROUP --no-wait --yes

疑難排解

如果您在本練習中遇到問題，請嘗試下列步驟：

容器應用程式沒有回應

  • 使用下列命令檢查修訂是否為作用中： az containerapp revision list
  • 使用下列命令確認輸入是否已設定： az containerapp show

無法看到記錄

  • 主控台記錄只會顯示近期資料。請使用 Log Analytics 檢視歷史資料。
  • Log Analytics 資料可能需要 2 到 5 分鐘才會顯示。

環境變數未生效

  • 當您變更環境變數時，Container Apps 會建立新的修訂。請確認新修訂為作用中。
  • 請謹慎使用 --replace-env-vars —它會取代所有環境變數，而不只是您指定的那些。
