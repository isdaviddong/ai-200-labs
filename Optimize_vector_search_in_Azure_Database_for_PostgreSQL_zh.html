最佳化 Azure Database for PostgreSQL 中的向量搜尋效能

在本練習中，您將部署 Azure Database for PostgreSQL 執行個體，並將其最佳化以支援向量搜尋工作負載。您會建立帶有向量嵌入的測試資料、分析基準效能、建置並比較 IVFFlat 與 HNSW 索引，並調整搜尋參數。對於需要在大型資料集上快速執行相似度搜尋的正式環境 AI 應用程式而言，這些技術至關重要。

本練習將執行的工作：

  • 下載專案起始檔案並設定部署指令碼
  • 部署使用 Microsoft Entra 驗證的 Azure Database for PostgreSQL Flexible Server
  • 建立含向量嵌入的測試資料集
  • 分析未使用索引時的向量搜尋基準效能
  • 建立並比較 IVFFlat 與 HNSW 向量索引
  • 調整索引參數以平衡速度與召回率

本練習約需 30 分鐘完成。

開始之前

若要完成本練習，您需要：

  • 具備部署所需 Azure 服務權限的 Azure 訂用帳戶。如果您尚未擁有，您可以 註冊一個。
  • Visual Studio Code 安裝於其中一個 支援的平台。
  • 最新版的 Azure CLI。
  • PostgreSQL command-line tools (psql)

檢閱向量索引概念

在本節中，您將檢閱稍後會在練習中實際應用的向量索引關鍵概念。理解這些取捨有助於您在最佳化向量搜尋時做出更明智的決策。

IVFFlat 索引

IVFFlat（Inverted File with Flat compression）會將向量分成稱為 lists的叢集。搜尋時，它只會掃描附近叢集中的向量，而不是整個資料集。

主要參數：

  • lists：要建立的叢集數。對於最多 100 萬筆資料，良好的起始值為 rows / 1000 。lists 越多，搜尋越快，但索引建置越慢。
  • probes：查詢時要搜尋的叢集數。值越高，召回率（找出真正最近鄰的能力）越好，但延遲也會增加。

HNSW 索引

HNSW（Hierarchical Navigable Small World）會建立多層圖形結構。上層包含較少節點以利快速導覽；下層包含較多節點以利精準搜尋。

主要參數：

  • m：每個節點的最大連線數。值越高可提升召回率，但也會增加記憶體使用量與建置時間。預設值為 16。
  • ef_construction：索引建置期間動態候選清單的大小。值越高可建立品質更佳的圖形，但建置時間也更長。預設值為 64。
  • ef_search：搜尋期間動態候選清單的大小。值越高可提升召回率，但也會增加延遲。預設值為 40。

何時使用各者

考量項目 IVFFlat HNSW
建置時間 較快 較慢
查詢速度 快速 較快
記憶體使用量 較低 較高
召回率準確性 調整後表現良好 預設情況更佳
更新效能 需要重建 支援增量更新

在本練習中，您將實際測試兩種索引類型，並親自衡量其取捨。

建立資料庫結構描述與測試資料

在本節中，您將連線到 PostgreSQL 伺服器，並建立一個包含產品資料與向量嵌入、供測試使用的資料表。

  1. 執行下列命令以使用環境變數連線到伺服器。 PGPASSWORD 環境變數會自動用於驗證。

    Bash

    psql "host=$DB_HOST port=5432 dbname=$DB_NAME user=$DB_USER sslmode=require"

    PowerShell

    psql "host=$env:DB_HOST port=5432 dbname=$env:DB_NAME user=$env:DB_USER sslmode=require"

    Tip: 顯示查詢結果時，如果目前終端機視窗無法容納全部內容，psql 會使用分頁器。若發生此情況，請按 q 以退出分頁器並返回 psql 提示字元。將終端機視窗放大可減少這種情況發生，也能更輕鬆檢閱命令結果。

  2. 執行下列命令以啟用 pgvector 擴充功能。PostgreSQL 擴充功能必須先明確啟用才能使用。pgvector 擴充功能新增了 vector 資料型別，以及像 <=> （cosine distance）這類您會在本練習中持續使用的運算子。Azure Database for PostgreSQL 內含 pgvector，但預設不會啟用。

    CREATE EXTENSION IF NOT EXISTS vector;

  3. 執行下列命令以建立含有向量欄位的 products 資料表。 vector(384) 資料型別可儲存 384 維嵌入向量，這是 sentence embedding 模型常見的大小。

    CREATE TABLE products (
    id BIGSERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    category_id INTEGER NOT NULL,
    price NUMERIC(10,2) NOT NULL,
    in_stock BOOLEAN DEFAULT true,
    embedding vector(384)
);

  4. 執行下列命令以產生含有隨機嵌入向量的測試資料。這會建立 100,000 個具有隨機 384 維向量的產品。

    INSERT INTO products (name, category_id, price, in_stock, embedding)
SELECT
    'Product ' || i,
    (random() * 20)::int + 1,
    (random() * 1000)::numeric(10,2),
    random() > 0.1,
    ('[' || array_to_string(ARRAY(
        SELECT (random() * 2 - 1)::float4
        FROM generate_series(1, 384)
    ), ',') || ']')::vector
FROM generate_series(1, 100000) AS i;

  5. 執行下列命令以確認資料已建立。您應該會看到 100,000 筆資料列。

    SELECT COUNT(*) FROM products;

  6. 執行下列命令以建立供一致性測試使用的查詢向量。這個暫存資料表會儲存您在整個練習中使用的隨機嵌入向量。

    CREATE TEMP TABLE query_vectors AS
SELECT ('[' || array_to_string(ARRAY(
    SELECT (random() * 2 - 1)::float4
    FROM generate_series(1, 384)
), ',') || ']')::vector AS embedding;

分析基準效能

在本節中，您將量測沒有任何索引時的向量搜尋效能，以建立基準。

  1. 執行下列命令以執行向量相似度查詢並擷取執行計畫。

    EXPLAIN ANALYZE
SELECT id, name, embedding <=> (SELECT embedding FROM query_vectors) AS distance
FROM products
ORDER BY embedding <=> (SELECT embedding FROM query_vectors)
LIMIT 10;

  2. 檢查輸出內容。您應該會看到 Seq Scan 出現在計畫中，表示 PostgreSQL 正在掃描全部 100,000 筆資料列。請記下底部的 Execution Time 值。

  3. 再執行此查詢兩次，以取得較一致的量測結果。第一次執行可能因為快取尚未預熱而較慢。請將最後一次的執行時間記錄為您的基準值。

建立並比較 IVFFlat 與 HNSW 索引

在本節中，您將建立這兩種索引類型並比較其效能。

建立 IVFFlat 索引

  1. 執行下列命令以建立 IVFFlat 索引。對於 100,000 筆資料列而言，100 個 lists 是合理的起始值（依據 rows / 1000 準則）。請記下建置索引所花的時間。

    CREATE INDEX idx_products_embedding_ivfflat
ON products USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);

  2. 執行下列命令以在 IVFFlat 索引下執行相同的查詢。確認計畫顯示 Index Scan using idx_products_embedding_ivfflat。請記錄執行時間。

    EXPLAIN ANALYZE
SELECT id, name, embedding <=> (SELECT embedding FROM query_vectors) AS distance
FROM products
ORDER BY embedding <=> (SELECT embedding FROM query_vectors)
LIMIT 10;

  3. 執行下列命令以使用較低的 probes 進行測試。這樣會更快，但因為只搜尋一個叢集，與完整掃描相比可能會漏掉真正的最近鄰。召回率的影響無法從時間輸出中直接看出，您需要比較實際結果才能衡量。

    SET ivfflat.probes = 1;
EXPLAIN ANALYZE
SELECT id, name, embedding <=> (SELECT embedding FROM query_vectors) AS distance
FROM products
ORDER BY embedding <=> (SELECT embedding FROM query_vectors)
LIMIT 10;

  4. 執行下列命令以使用較高的 probes 進行測試（較慢、較高召回率）。請記錄執行時間。

    SET ivfflat.probes = 50;
EXPLAIN ANALYZE
SELECT id, name, embedding <=> (SELECT embedding FROM query_vectors) AS distance
FROM products
ORDER BY embedding <=> (SELECT embedding FROM query_vectors)
LIMIT 10;

建立 HNSW 索引

  1. 執行下列命令以刪除 IVFFlat 索引，讓您可以獨立測試 HNSW。

    DROP INDEX idx_products_embedding_ivfflat;

  2. 執行下列命令以增加索引建置可用的記憶體。HNSW 在建置期間比 IVFFlat 需要更多記憶體。

    SET maintenance_work_mem = '256MB';

  3. 執行下列命令以建立 HNSW 索引。請記下建置時間，通常會比 IVFFlat 更長。

    CREATE INDEX idx_products_embedding_hnsw
ON products USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);

  4. 執行下列命令以在 HNSW 索引下執行查詢。

    EXPLAIN ANALYZE
SELECT id, name, embedding <=> (SELECT embedding FROM query_vectors) AS distance
FROM products
ORDER BY embedding <=> (SELECT embedding FROM query_vectors)
LIMIT 10;

  5. 執行下列命令以使用較低的 ef_search 進行測試。這樣會更快，但因為搜尋探索的候選路徑較少，可能會漏掉部分真正的最近鄰。召回率的影響不會顯示在時間輸出中。

    SET hnsw.ef_search = 20;
EXPLAIN ANALYZE
SELECT id, name, embedding <=> (SELECT embedding FROM query_vectors) AS distance
FROM products
ORDER BY embedding <=> (SELECT embedding FROM query_vectors)
LIMIT 10;

  6. 執行下列命令以使用較高的 ef_search 進行測試（較高召回率）。請記錄執行時間。

    SET hnsw.ef_search = 100;
EXPLAIN ANALYZE
SELECT id, name, embedding <=> (SELECT embedding FROM query_vectors) AS distance
FROM products
ORDER BY embedding <=> (SELECT embedding FROM query_vectors)
LIMIT 10;

比較您的結果

比較不同設定下的執行時間。您應該會觀察到：

  • Sequential scan 最慢，因為它會檢查全部 100,000 筆資料列
  • IVFFlat with probes=1 是最快的索引化選項，但可能會漏掉部分真正的最近鄰
  • HNSW 在相近的召回率下，通常能提供比 IVFFlat 更快的查詢
  • 提高 probes （IVFFlat）或 ef_search （HNSW）可提升準確度，但也會增加延遲

使用索引實作中繼資料篩選

在本節中，您將測試結合向量相似度與中繼資料篩選的查詢。在正式環境應用程式中，純向量搜尋並不常見；您通常會先依類別、日期範圍、價格或其他屬性進行篩選，再尋找相似項目。要最佳化這類複合查詢，就必須理解 PostgreSQL 如何同時使用多種索引類型。

  1. 執行下列命令以在 category 欄位上建立 B-tree 索引。

    CREATE INDEX idx_products_category ON products (category_id);

  2. 執行下列命令以執行帶有篩選條件的向量搜尋。檢查執行計畫，您應該會看到 PostgreSQL 先使用 HNSW 索引處理向量相似度，再套用 category 篩選。請記下執行時間。

    EXPLAIN ANALYZE
SELECT id, name, embedding <=> (SELECT embedding FROM query_vectors) AS distance
FROM products
WHERE category_id = 5
ORDER BY embedding <=> (SELECT embedding FROM query_vectors)
LIMIT 10;

  3. 執行下列命令以用更具選擇性的篩選條件進行測試。當有多個篩選條件時，您可能會看到 Bitmap Index Scan 出現在 B-tree 索引上並搭配後續篩選，或者規劃器可能會選擇不同策略。請將執行時間與前一個查詢比較。

    EXPLAIN ANALYZE
SELECT id, name, embedding <=> (SELECT embedding FROM query_vectors) AS distance
FROM products
WHERE category_id = 5 AND price BETWEEN 100 AND 200
ORDER BY embedding <=> (SELECT embedding FROM query_vectors)
LIMIT 10;

  4. 執行下列命令以為此篩選組合建立複合索引。

    CREATE INDEX idx_products_category_price ON products (category_id, price);

  5. 重新執行前一個查詢並比較執行計畫。複合索引可能讓 PostgreSQL 在向量搜尋之前或同時更有效率地篩選，進而降低執行時間。

    EXPLAIN ANALYZE
SELECT id, name, embedding <=> (SELECT embedding FROM query_vectors) AS distance
FROM products
WHERE category_id = 5 AND price BETWEEN 100 AND 200
ORDER BY embedding <=> (SELECT embedding FROM query_vectors)
LIMIT 10;

疑難排解

如果您在本練習期間遇到問題，請嘗試下列步驟：

psql 連線失敗

  • 確認 .env 檔案已透過執行部署指令碼選項 4 建立
  • 確認您已執行 source .env （Bash）或 . ..env.ps1 （PowerShell）以載入環境變數
  • 存取權杖約在一小時後過期；請再次執行部署指令碼選項 4 以產生新的權杖
  • 執行部署指令碼選項 3 來確認伺服器已就緒

存取遭拒或驗證錯誤

  • 確認已透過執行部署指令碼選項 2 設定 Microsoft Entra 系統管理員
  • 確認 PGPASSWORD 已在您的終端機工作階段中正確設定
  • 確認您使用正確的 DB_USER 值（您的 Azure 帳戶電子郵件）

索引建置耗時過久或失敗

  • HNSW 索引的建置時間比 IVFFlat 更長；對於 100,000 個向量，請預留 1 到 2 分鐘
  • 如果建置逾時，請檢查 Azure Monitor 中的 CPU 與記憶體計量
  • 請考慮縮小資料集大小以進行測試
