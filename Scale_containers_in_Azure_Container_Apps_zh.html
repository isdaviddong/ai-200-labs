使用 KEDA 觸發程序設定自動調整

AI 應用程式經常會遇到不可預測的工作負載，例如推論要求激增、批次作業，或來自代理程式式工作流程的突然尖峰。Azure Container Apps 中以 KEDA 為基礎的自動調整可讓您的工作負載在閒置時縮減為零（節省成本），並在需求增加時快速向外擴展。

在本練習中，您會部署一個簡單的模擬代理程式 API，並根據HTTP 並行要求設定自動調整。接著，您會產生並行負載，並觀察應用程式如何在套用設定變更時向外擴展並建立新的修訂。

本練習中將完成下列工作：

  • 建立 Azure Container Registry 和 Container Apps 資源
  • 部署模擬代理程式 API 容器應用程式
  • 使用 KEDA 設定 HTTP 並行調整規則
  • 產生並行要求以觸發向外擴展，並即時監視複本數量變化
  • 使用 YAML 設定調整規則

本練習約需 30 分鐘完成。

重要： Azure Container Registry 作業執行 目前暫不支援 Azure 免費額度。本練習需要 Pay-As-You-Go 或其他付費方案。

開始之前

若要完成本練習，您需要：

  • 具備部署所需 Azure 服務權限的 Azure 訂閱。如果您還沒有，可以 註冊一個.
  • Visual Studio Code ，可在下列其中一個 支援的平台上執行.
  • 最新版的 Azure CLI.
  • Python 3.12 或更新版本。

下載專案起始檔並部署 Azure 服務

在本節中，您會下載專案起始檔，並使用指令碼將所需服務部署到您的 Azure 訂閱。Azure Container Registry 與 Container Apps 環境的部署需要幾分鐘才能完成。

  1. 開啟瀏覽器並輸入下列 URL 以下載起始檔。檔案會儲存在您的預設下載位置。

    https://github.com/MicrosoftLearning/mslearn-azure-ai/raw/main/downloads/python/aca-scale-python.zip

  2. 將檔案複製或移動到系統中您希望進行專案作業的位置。然後將檔案解壓縮到資料夾中。

  3. 啟動 Visual Studio Code (VS Code)，並在功能表中選取 File > Open Folder...，然後選擇包含專案檔案的資料夾。

  4. 此專案同時包含 Bash（azdeploy.sh）與 PowerShell（azdeploy.ps1）的部署指令碼。開啟適合您環境的檔案，將指令碼頂端的兩個值修改為符合您的需求，然後儲存變更。注意： 不要變更指令碼中的其他任何內容。

    "<your-resource-group-name>" # Resource Group name
"<your-azure-region>" # Azure region for the resources

  5. 在功能表列中選取 Terminal > New Terminal，在 VS Code 中開啟終端機視窗。

  6. 執行下列命令登入您的 Azure 帳戶。依照提示選取本練習要使用的 Azure 帳戶與訂閱。

    az login

  7. 執行下列命令，以確認您已安裝 Azure CLI 的 containerapp 延伸模組。

    az extension add --name containerapp

  8. 執行下列命令，以確保您的訂閱具有本練習所需的資源提供者。

    az provider register --namespace Microsoft.App
az provider register --namespace Microsoft.OperationalInsights
az provider register --namespace Microsoft.ContainerRegistry

在 Azure 中建立資源

在本節中，您會執行部署指令碼，將所需服務部署到您的 Azure 訂閱。

  1. 請確認您位於專案根目錄，並在終端機中執行對應命令來啟動部署指令碼。此指令碼會部署 ACR、Container Apps 環境，以及已啟用輸入的 Container App。它也會建立一個包含環境變數的檔案，供您在整個練習中使用。

    Bash

    bash azdeploy.sh

    PowerShell

    ./azdeploy.ps1

    注意： 如果 PowerShell 因為指令碼未經數位簽署而封鎖它，請在同一個終端機工作階段中執行下列命令，然後再次執行部署指令碼。此命令只會變更目前 PowerShell 處理序的執行原則。

    Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass

  2. 當指令碼執行時，輸入 1 以啟動 Create Azure Container Registry and build container image。

  3. 當前一個作業完成後，輸入 2 以啟動 Create Container Apps environment。

  4. 當前一個作業完成後，輸入 3 以啟動 Create Container App。

    注意： 容器應用程式建立後，系統會建立一個包含環境變數的檔案。您會在整個練習中使用這些變數。

  5. 當部署完成後，輸入 5 以結束部署指令碼。

  6. 執行對應命令，從先前步驟建立的檔案中將環境變數載入目前的終端機工作階段。

    Bash

    source .env

    PowerShell

    . .\.env.ps1

  7. 確認應用程式端點可供使用。

    Bash

    curl -sS "$CONTAINER_APP_URL/" | head

    PowerShell

    Invoke-RestMethod "$env:CONTAINER_APP_URL/"

    注意： 請保持此終端機開啟。如果您將它關閉並建立新的終端機，可能需要再次執行命令來建立環境變數。

設定自動調整

在本節中，您會設定一個 HTTP 調整規則，依據並行要求來觸發調整。這是「進行中的代理程式要求」的實用代理指標，而且不需要新增其他 Azure 服務。

注意： 套用設定更新（包括調整變更）會建立新的修訂。

  1. 執行下列命令，以使用 HTTP 調整規則更新容器應用程式。此規則會監視進行中的並行要求，並在需求增加時調整應用程式。

    Bash

    az containerapp update \
    --name $CONTAINER_APP_NAME \
    --resource-group $RESOURCE_GROUP \
    --min-replicas 0 \
    --max-replicas 10 \
    --scale-rule-name http-scaling \
    --scale-rule-type http \
    --scale-rule-http-concurrency 10

    PowerShell

    az containerapp update `
    --name $env:CONTAINER_APP_NAME `
    --resource-group $env:RESOURCE_GROUP `
    --min-replicas 0 `
    --max-replicas 10 `
    --scale-rule-name http-scaling `
    --scale-rule-type http `
    --scale-rule-http-concurrency 10

  2. 執行下列命令以確認已設定調整規則。在輸出中尋找 http-scaling 規則，其中 minReplicas 設為 0，而 maxReplicas 設為 10。

    Bash

    az containerapp show \
    --name $CONTAINER_APP_NAME \
    --resource-group $RESOURCE_GROUP \
    --query "properties.template.scale"

    PowerShell

    az containerapp show `
    --name $env:CONTAINER_APP_NAME `
    --resource-group $env:RESOURCE_GROUP `
    --query "properties.template.scale"

產生負載並觀察調整

在本節中，您會執行本機 Flask 儀表板，它可以同時產生並行要求，並顯示 Container App 的修訂與複本。

  1. 執行下列命令以移至 client 目錄。

    cd client

  2. 執行下列命令，為用戶端應用程式建立虛擬環境。視您的環境而定，命令可能是 python 或 python3。

    python -m venv .venv

  3. 執行下列命令以啟用 Python 環境。注意： 在 Linux/macOS 上，請使用 Bash 命令。在 Windows 上，請使用 PowerShell 命令。如果您在 Windows 上使用 Git Bash，請使用 source .venv/Scripts/activate。

    Bash

    source .venv/bin/activate

    PowerShell

    .\.venv\Scripts\Activate.ps1

  4. 執行下列命令以安裝用戶端應用程式的相依性。

    pip install -r requirements.txt

  5. 執行下列命令以啟動儀表板。

    python app.py

  6. 開啟瀏覽器並瀏覽至下列 URL：http://127.0.0.1:5000。

  7. 在應用程式左側窗格中選取 Refresh Revisions & Replicas。在應用程式右上角，您應該會看到有 1 或 0 個複本正在執行。

    部署應用程式時，預設會有 1 個複本執行。您已在前一個步驟中套用 KEDA 調整規則，而由於預設的 300 秒（5 分鐘） 冷卻期間，當工作負載進入閒置後，縮減到零可能還需要額外的 約 5 分鐘。

  8. 在 Load Generator 區段中，選取 Start 開始將資料傳送到容器應用程式。

  9. 每隔 5 到 10 秒選取一次 Refresh Revisions & Replicas，您應該會看到複本數量增加。在它停止後，您可以再次執行 Load Generator，以提高流量並增加複本數。

完成後，請關閉瀏覽器視窗，並在終端機中輸入 Ctrl+c 以結束用戶端應用程式。

使用 YAML 設定調整規則

在本節中，您會透過編輯 Container App YAML 來設定自動調整。這是一種可重複使用的調整規則管理方式，而且當您擁有多個規則時會變得格外重要。

  1. 執行下列命令，將應用程式設定匯出到 YAML 檔案。

    Bash

    az containerapp show \
    --name $CONTAINER_APP_NAME \
    --resource-group $RESOURCE_GROUP \
    --output yaml > app-config.yaml

    PowerShell

    az containerapp show `
    --name $env:CONTAINER_APP_NAME `
    --resource-group $env:RESOURCE_GROUP `
    --output yaml > app-config.yaml

  2. 在 VS Code 中開啟 app-config.yaml 檔案。在 properties > template 下方找到 scale 區段。修改調整設定，將 cooldownPeriod 降為 200 秒（更快縮減）、將 maxReplicas 設為 5，並將 minReplicas 設為 1，使應用程式至少始終有一個複本在執行。scale 區段應與下列範例類似。

    scale:
  cooldownPeriod: 200
  maxReplicas: 5
  minReplicas: 1
  pollingInterval: 30

  3. 儲存檔案，然後執行下列命令以套用更新後的設定。

    Bash

    az containerapp update \
    --name $CONTAINER_APP_NAME \
    --resource-group $RESOURCE_GROUP \
    --yaml app-config.yaml

    PowerShell

    az containerapp update `
    --name $env:CONTAINER_APP_NAME `
    --resource-group $env:RESOURCE_GROUP `
    --yaml app-config.yaml

  4. 執行下列命令以確認您剛剛實作的變更。

    Bash

    az containerapp show \
    --name $CONTAINER_APP_NAME \
    --resource-group $RESOURCE_GROUP \
    --query "properties.template.scale"

    PowerShell

    az containerapp show `
    --name $env:CONTAINER_APP_NAME `
    --resource-group $env:RESOURCE_GROUP `
    --query "properties.template.scale"

清理資源

現在您已完成練習，應刪除所建立的雲端資源，以避免不必要的資源使用。

  1. 在 VS Code 終端機中執行下列命令，以刪除資源群組及群組中的所有資源。請將 <rg-name> 取代為您在本練習稍早選擇的名稱。此命令會在 Azure 中啟動背景工作來刪除資源群組。

    az group delete --name <rg-name> --no-wait --yes

注意： 刪除資源群組會刪除其中包含的所有資源。如果您為本練習選擇的是現有資源群組，則本練習範圍外的任何既有資源也會一併遭到刪除。

疑難排解

如果您在本練習期間遇到問題，請嘗試下列步驟：

應用程式在負載下未向外擴展

  • 確認已設定 HTTP 調整規則：az containerapp show --query "properties.template.scale"
  • 確認您正在產生並行要求（使用 delayMs > 0 的儀表板）
  • 提高 delayMs（500-1500ms），讓要求彼此重疊並累積並行數
  • 檢查系統記錄中的調整事件：az containerapp logs show --type system --tail 50

儀表板無法啟動或無法列出修訂／複本

  • 確認已啟用 Python 虛擬環境（您應該會在終端機提示字元中看到 (.venv)）
  • 確認已安裝相依性：pip install -r client/requirements.txt
  • 確認已安裝 Azure CLI 並執行過 az login
  • 確認已安裝 containerapp 延伸模組：az extension add --name containerapp
  • 確認已載入 .env，而且其中包含 RESOURCE_GROUP 與 CONTAINER_APP_NAME

Python venv 啟用問題

  • 在 Linux/macOS 上，請使用：source client/.venv/bin/activate
  • 在 Windows PowerShell 上，請使用：.\client.venv\Scripts\Activate.ps1
  • 如果缺少 activate 指令碼，請重新安裝 python3-venv 套件並重新建立 venv

YAML 更新失敗

  • 確認 YAML 檔案語法有效（檢查縮排）
  • 某些唯讀屬性，例如 id、systemData 和 type，可能會造成錯誤；如有需要請將其移除
  • 確認 scale 區段在 properties > template > scale 下方遵循正確結構
